# v3.15.14 ‚Äî Sprint-aware COL Routing

Date: 2026-04-27
Branch: `feature/v3.15.14-sprint-aware-col-routing`
VERSION bump: `3.15.13 ‚Üí 3.15.14`

## Scope

v3.15.13 shipped the Discovery Sprint Orchestrator as **observer-only**.
A sprint registry exists, the orchestrator counts matching completed
campaigns, but the v3.15.2 Campaign Operating Layer (COL) does not
read the registry ‚Äî so an active sprint produces zero progress
unless COL happens to spawn matching presets autonomously. v3.15.14
closes that loop.

When a sprint is **active** and its window has not expired and its
target has not been met, the campaign launcher filters its candidate
set down to the sprint's plan preset names **before** invoking
`campaign_policy.decide()`. The policy engine itself is unchanged ‚Äî
the v3.15.11 regression pin (no advisory artifacts in `decide()`'s
signature; same inputs ‚Üí byte-identical artifact) is preserved
verbatim. Routing happens by shrinking the input, not by extending
the engine.

When no sprint is active (or its registry says `completed`,
`expired`, or `canceled`), the launcher's behavior is **identical to
v3.15.13** ‚Äî full template catalog, full followup/control specs,
unchanged tie-break ordering.

## Modules

### `research/discovery_sprint.py` (extended)

- `ActiveSprintConstraints` (frozen) ‚Äî `(sprint_id, profile_name,
  plan_preset_names: frozenset[str], plan_hypothesis_ids:
  frozenset[str], target_campaigns, started_at_utc,
  expected_completion_at_utc, observed_total)`.
- `load_active_sprint_constraints(*, campaign_registry=None,
  now_utc=None, sprint_registry_path=None) -> ActiveSprintConstraints |
  None`. Read-only. Returns `None` when:
  - sprint registry artifact missing or malformed,
  - `state` ‚âÝ `"active"` (covers `completed`, `expired`,
    `canceled`),
  - `now_utc >= expected_completion_at_utc`,
  - `campaign_registry` supplied AND `observed_total >=
    target_campaigns`.
- `apply_sprint_routing(*, templates, follow_up_specs,
  weekly_control_specs, sprint_constraints) -> (templates, followups,
  controls, counts)`. Pure. When constraints is `None`, passes through
  unchanged AND emits `templates_filtered = templates_total` so the
  caller always sees consistent shape.
- `build_routing_decision_payload(...)` + `write_routing_decision_artifact(...)`
  ‚Äî read-only audit sidecar
  `research/discovery_sprints/sprint_routing_decision_latest.v1.json`.
  Carries `routing_active`, sprint snapshot, original/filtered counts,
  and the chosen `decision.action / preset_name / template_id /
  reason`.
- `sprint_extra_for_record(constraints) -> dict` ‚Äî returns
  `{"sprint_id": ..., "sprint_profile_name": ..., "sprint_routing":
  "v3.15.14"}` when a sprint is active, else `{}`.
- New constants: `SPRINT_ROUTING_DECISION_PATH`,
  `SPRINT_RECORD_EXTRA_KEYS`, `INACTIVE_SPRINT_STATES`
  (`{"completed", "expired", "canceled"}`).

### `research/campaign_launcher.py` (modified)

In `_tick`, between `_build_*_specs` and the call to `decide()`:

```python
sprint_constraints = load_active_sprint_constraints(
    campaign_registry=registry,
    now_utc=now_utc,
)
(templates_for_decide,
 followup_specs_for_decide,
 control_specs_for_decide,
 routing_counts) = apply_sprint_routing(
    templates=templates,
    follow_up_specs=followup_specs,
    weekly_control_specs=control_specs,
    sprint_constraints=sprint_constraints,
)
decision = decide(
    ...,
    templates=templates_for_decide,
    follow_up_candidate_specs=followup_specs_for_decide,
    weekly_control_candidate_specs=control_specs_for_decide,
    ...,
)
```

After the decision is taken, when `sprint_constraints is not None`:

```python
write_routing_decision_artifact(build_routing_decision_payload(...))
```

The sidecar write is wrapped in a `try/except OSError` ‚Äî purely
diagnostic, never fails a tick.

In `_apply_decision`, the spawn branch now passes
`extra=sprint_extra_for_record(sprint_constraints)` to `_build_record`,
which stamps `CampaignRecord.extra` with `sprint_id /
sprint_profile_name / sprint_routing`. `extra` is an existing
`dict[str, Any]` field ‚Äî additive nullable, no schema break.

`_build_record` gained an `extra` keyword.

### `docker-compose.yml` (modified ‚Äî mandatory deploy fix)

Added `./research:/app/research` to the `agent` service's volumes.
Without this mount the launcher (which runs inside the `agent`
container) cannot read the sprint registry written via the
`dashboard` container, and routing silently remains inactive on the
VPS ‚Äî defeating the whole release.

`apply_sprint_routing()` is a pure filter ‚Äî feeding it `None`
constraints is equivalent to v3.15.13 behavior, so the launcher path
collapses cleanly when a sprint terminates mid-tick.

## Traceability

Three independent surfaces emit "sprint routing was active":

1. **Per-tick sidecar** ‚Äî
   `research/discovery_sprints/sprint_routing_decision_latest.v1.json`.
   Read-only, atomic, deterministic. Carries the active sprint
   snapshot + filtered counts + chosen decision.
2. **Spawned `CampaignRecord.extra`** ‚Äî every campaign queued during
   a sprint tick carries `sprint_id` and `sprint_profile_name` in its
   `extra` dict, persisted in `campaign_registry_latest.v1.json`. This
   lets backward-looking queries (e.g. observability dashboards or
   the future v3.15.15 reporter) correlate completed campaigns to
   the sprint that drove them.
3. **`discovery_sprint status`** ‚Äî unchanged from v3.15.13. Counts
   `campaign_registry` records whose `preset_name ‚àà plan` and
   `finished_at_utc ‚àà [started_at, now]`. With v3.15.14 wiring,
   those counts now grow as COL spawns matching campaigns.

## Duplicate prevention

The campaign policy engine's `_filter_candidate` already enforces:

- preset cooldown (`time_since_last < effective_cooldown` ‚Äî daily
  primaries default 86400s);
- registry duplicate check (`has_duplicate(...)` keyed on
  `(campaign_type, preset_name, parent_or_lineage_root,
  input_artifact_fingerprint)`);
- single-worker admission (`max_concurrent_campaigns`).

These all fire identically when a sprint is active. v3.15.14 adds no
new duplicate logic ‚Äî sprint routing only changes which presets get
to enter the candidate set.

## Known limitations (documented, not fixed in v3.15.14)

- `vol_compression_breakout_crypto_1h` is in `research/presets.py`
  (since v3.15.4) and is bound to `volatility_compression_breakout_v0`
  in the hypothesis catalog (status `active_discovery`), but the
  preset was never added to `_HYPOTHESIS_AWARE_PRESETS` in
  `research/campaign_templates.py`. As a result, `CAMPAIGN_TEMPLATES`
  has no `daily_primary` / `survivor_confirmation` / `paper_followup`
  / `weekly_retest` / `daily_control` row for it. A sprint plan still
  *includes* this preset (correct, per the source-of-truth in PRESETS
  + catalog), but the launcher cannot spawn what has no template, so
  the routing helper's `templates_filtered` count for this preset is
  zero in production today.

  Net effect: while the gap exists, sprint progress accrues solely
  from `trend_pullback_crypto_1h`. The routing layer still emits the
  correct sidecar + extra metadata; nothing spurious fires; the
  observer count is honest. Closing the gap is a separate
  v3.15.4-style mechanical fix ‚Äî it adds **template config** for an
  already-existing preset/strategy, not a new strategy. Out of scope
  for v3.15.14 per the user's "Use existing campaign templates"
  constraint.

## Out of scope (deferred)

- Frontend exposure of `sprint_routing_decision_latest.v1.json`
  (per the prompt's "Do not create frontend business logic"
  constraint).
- The `vol_compression_breakout_crypto_1h` template gap (see
