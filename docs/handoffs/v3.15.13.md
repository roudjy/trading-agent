# v3.15.13 — Discovery Sprint Orchestrator (artifact-only)

Date: 2026-04-27
Branch: `feature/v3.15.13-discovery-sprint-orchestrator`
VERSION bump: `3.15.12 → 3.15.13`

## Scope

Adds a **bounded observation-window controller** layered on top of the
v3.15.2 Campaign Operating Layer (COL). A *discovery sprint* is a
profile-driven, time-boxed measurement: it snapshots a closed profile
plus a deterministic plan, then tracks how many COL-completed
campaigns matching the plan land within `[started_at,
started_at + max_days]`.

Hard positioning (no exceptions):

- **Artifact-only.** No queue, registry, lease, ledger, or frozen
  contract is ever mutated. There is no spawn surface; the
  orchestrator cannot bypass COL.
- **Idempotent `run`.** Refuses to start a second sprint while another
  is `state="active"` and within its window.
- **Deterministic `plan`.** Same inputs → byte-identical plan output.
- **Closed built-in profile registry.** This release ships exactly
  one: `crypto_exploratory_v1`. New profiles must be added by editing
  `BUILTIN_PROFILES`, never via runtime config.

## Modules

- `research/discovery_sprint.py` — pure profile + plan + payload
  builders, thin IO wrappers, CLI dispatcher.
  - `SprintProfile` (frozen dataclass) + `BUILTIN_PROFILES` closed
    dict with `crypto_exploratory_v1`.
  - `derive_plan(profile)` filters preset catalog on `hypothesis_id ∈
    profile.hypotheses`, `timeframe ∈ profile.timeframes`,
    `screening_phase == profile.screening_phase`, `enabled` and
    `status == "stable"`, plus inferred `asset_class` (universe
    suffix `-EUR/USD/USDT/USDC/BUSD` ⇒ crypto, else equity, mixed
    rejected).
  - `count_observations(...)` reads the v3.15.2 campaign registry
    (read-only) and counts `state=="completed"` records whose
    `preset_name ∈ plan` and `finished_at_utc ∈ window`.
  - `compute_sprint_id()` — deterministic
    `sprt-<utc_compact>-<sha256[:10]>` over canonical(profile +
    started_at).
  - CLI: `plan`, `run`, `status`, `report` (see below).

No other module is touched in v3.15.13. `research/run_research.py`,
`research/campaign_launcher.py`, and `research/campaign_policy.decide()`
are unchanged.

## CLI

```
python -m research.discovery_sprint plan   --profile crypto_exploratory_v1
python -m research.discovery_sprint run    --profile crypto_exploratory_v1
python -m research.discovery_sprint status
python -m research.discovery_sprint report
```

Exit codes:

- `0` success.
- `1` refusal (active sprint already exists for `run`; sprint still
  active for `report`).
- `2` validation error (unknown profile, empty plan, missing
  timestamps in registry artifact).

## Artifacts

Directory: `research/discovery_sprints/` (created on first `run`).

- `sprint_registry_latest.v1.json` — bounded sprint state. Fields:
  `sprint_id`, `state ∈ {active, completed, expired}`, `started_at_utc`,
  `expected_completion_at_utc`, `completed_at_utc`, full
  `profile` snapshot, full `plan.entries[]`. Carries the standard COL
  pin block (`live_eligible: false`, `authoritative: false`,
  `diagnostic_only: true`).
- `discovery_sprint_progress_latest.v1.json` — counts at last
  `status` invocation. Fields include `target_campaigns`,
  `observed_total`, `pct_complete`, `days_remaining`, `target_met`,
  `expired`, `by_hypothesis`, `by_preset`, `by_outcome`.
- `discovery_sprint_report_latest.v1.json` — written by `report`
  only when registry state is `completed` or `expired`. Mirrors the
  registry header plus an `outcome_summary` block with counts and
  shortfall codes.

All artifacts are written through `research._sidecar_io.write_sidecar_atomic`
(canonical JSON, deterministic key ordering, atomic rename).

## State machine

```
            run (no active sprint)               status (target_met)
   absent ────────────────────────► active ────────────────────────► completed
                                      │
                                      │ status (now ≥ expected_completion_at)
                                      └─────────────────────────────► expired

   report:
     - state == active        → exit 1 (refused)
     - state ∈ {completed, expired} → write report artifact, exit 0
     - registry absent        → exit 2
```

`status` is the only handler that may transition `active → completed`
or `active → expired`. It always re-counts observations from the
campaign registry and rewrites the progress artifact.

## Tests

`tests/unit/test_discovery_sprint.py` — 27 tests:

- profile registry contents + structural validation (zero target,
  unknown hypothesis, crypto-without-exclude_equities,
  exploratory-without-exclude_promotion_grade)
- plan determinism (byte-equal across calls), equities exclusion,
  non-exploratory exclusion, hypothesis allowlist, both target presets
  present, asset-class inference matrix
- sprint id determinism, `is_active_sprint` matrix
- CLI: plan emits + plan unknown-profile, run writes both artifacts,
  run refuses while active, status reports `no_sprint`, status counts
  observations from a fake registry, status transitions to
  `completed` (target met) and `expired` (window passed)
- report refuses while active, report writes after completion
- frozen contracts (`research_latest.json`, `strategy_matrix.csv`)
  unchanged across `plan` + `run` + `status`
- source-level guard: the orchestrator file does not call any COL
  mutator (`write_queue`, `write_registry`, `write_decision`,
  `append_events`, `transition_state`, `record_outcome`)

## Out of scope (deferred)

- Any policy-side consumer that biases COL toward the active sprint.
  Today the sprint observes the queue COL would have produced anyway.
  A future release may add a `campaign_policy` integration point that
  reads the sprint registry; that is a v3.16.x or later change with
  its own ADR.
- Multiple concurrent sprints, profile composition, or a configurable
  profile registry.
- Budget/cost integration with `campaign_budget`.
- `4h` crypto exploratory presets — adding presets is a separate
  decision and explicitly out of scope here.
