## 4. Architecturele keuzes (met motivatie)

- **Overlay-first blijft leidend**. v3.14 raakt géén v3.12 of v3.13
  artifact aan. Alle nieuwe state landt op vier nieuwe sidecars, elk
  met eigen `schema_version`-pin, joined op `candidate_id` of
  `sleeve_id`.
- **Runner dun**. `run_research.py` krijgt drie kleine, expliciete
  additions na de v3.13 hook — geen portfolio-logica inline, alles
  via de façade. Mirror exact van v3.13-patroon.
- **Cache-first width feed**. Optie A uit de v3.13 handoff is
  gekozen: `MarketRepository.get_bars` met identieke key als de
  backtest → cache-hit. Geen engine-contract uitbreiding, geen
  netwerk-download. Bij cache-miss valt de feed per pair netjes
  terug (`distributions[candidate_id]` wordt overgeslagen) en het
  v3.13-pad markeert dat axis gewoon `insufficient` — geen
  regressie.
- **Returns-bridge via public accessor**. `evaluations` wordt in
  `run_research.py` al accumulated; de bridge consumeert
  `engine.last_evaluation_report.evaluation_samples.daily_returns`
  die al public is via screening_runtime.py gebruikspatroon. Geen
  engine-wijziging nodig (Open Question A uit plan: resolved).
- **Sleeve key uit `experiment_family`**. Geen nieuwe veldderivatie —
  v3.12 registry v2 voert al `{strategy_family}|{asset_class}` als
  stabiele pre-computed string. Plus `interval`. Geen forced sleeves
  en geen verborgen assumptions (Open Question B uit plan:
  resolved).
- **Diagnostic-only by construction**. Elke threshold is een named
  constant + in artifact payload + expliciet warning-only +
  `authoritative=false` + `diagnostic_only=true` top-level. Geen
  consumer mag v3.14 output gebruiken voor promotion / paper / live
  decisions. Frozen scoring pin (`v0.1-experimental`,
  `composite_status="provisional"`, `authoritative=False`) blijft
  onveranderd.
- **Aligned-suffix correlatie**. Daily returns komen aggregated uit
  de engine zonder expliciete timestamp-array. Suffix-alignment is
  eerlijk met de data die we hebben en garandeert dat consumers over
  hetzelfde recente venster redeneren. Precision-upgrade via
  typed-timestamped stream is v3.15+.
- **Equal-weight only**. Vol-targeted en capped-concentration zijn
  expliciet uitgesteld. v3.14 laat zich niet verleiden tot een
  allocator-lookalike.

## 8. Bekende beperkingen (voor v3.15 aan te pakken)

### 8.1 Correlation uses suffix-alignment, not timestamp-alignment

Elke `CandidateReturnsRecord` draagt één geaggregeerde
`daily_returns` tuple, gesorteerd via engine-folds. Zonder
expliciete timestamp-array kan v3.14 alleen op **lengte** aligneren
(`min_len` vanaf het eind). Voor candidates met sterk divergerende
OOS-vensters is dat conservatief maar niet precies. Oplossing in
v3.15: typed timestamped returns stream (bijvoorbeeld per-fold
date-indexed), zodat correlatie echt per-bar-datum aligneert.

### 8.2 Joint trend/vol tagging ongewijzigd

De v3.13 §8.2 beperking is bewust niet aangepakt in v3.14: de
portfolio-diagnostics werken vanuit de returns-stream (correlatie,
DD-attribution, concentratie), niet vanuit de trend/vol joint
counts. Pas wanneer v3.15+ sleeve-gedrag splitst op regime-buckets
op bar-niveau wordt joint tagging strict necessary.

### 8.3 Filtered subset Sharpe/max-DD blijft null

Onveranderd uit v3.13 §8.3. Bar-stream serialisatie is het
natuurlijke v3.15+ volgende pad.

### 8.4 Scoring blijft v0.1-experimental

Bewust niet gebumped in v3.14. `regime_breadth_diagnostic` is
alleen op sleeve-niveau zichtbaar, nooit als composite component.
v3.15 kan `SCORING_FORMULA_VERSION` naar `v0.2-experimental` bumpen
(authoritative blijft `False`) met bijpassende goldens-update.

### 8.5 Equal-weight only

Volatility-targeted en capped-concentration sims deferred naar
v3.15+. De bestaande `portfolio_diagnostics_latest.v1.json.equal_weight_portfolio`
block is de natural slot waar nieuwe portfolio-varianten
toegevoegd kunnen worden zonder schema-break (via additive keys).

### 8.6 Geen frontend surface

Consumptie gaat via `/api/registry/portfolio` + markdown report.
v3.15+ kan een read-only sleeves/portfolio tab toevoegen zodra er
een concrete operationele behoefte is.

## 9. v3.15 startpunten — Paper Validation Engine

### Natuurlijke inputs vanuit v3.14

- **`candidate_returns_latest.v1.json`** is de kandidaat-seed voor
  paper-simulatie. Pre-computed daily returns per candidate_id met
  expliciete alignment en timestamp-semantics.
- **`portfolio_diagnostics_latest.v1.json.equal_weight_portfolio`**
  is de portfolio-baseline die de paper engine als referentie kan
  gebruiken voor ensemble-paper-flows.
- **`sleeve_registry_latest.v1.json`** is de natuurlijke
  paper-sleeve-grouping zonder hergroepering te hoeven ontwerpen.
- **`regime_width_distributions_latest.v1.json`** + v3.13 overlay
  geven de paper engine regime-bewust startpunt voor conditionele
  rules.

### Aanbevolen v3.15 scope (uit roadmap)

- Paper execution simulation (niet live).
- Typed timestamped returns stream als precision-upgrade voor
  correlatie en paper-trade attributie.
- Regime-filtered preset promotion op basis van schone v3.14/v3.13
  evidence — alleen als de breadth/concentration diagnostics het
  verantwoorden.
- Eventueel `SCORING_FORMULA_VERSION` bump naar `v0.2-experimental`
  met regime-breadth component (provisional, non-authoritative
  blijft).

### Wat v3.15 expliciet **niet** moet doen

- Live execution / broker integraties.
- Shadow mode / production allocator.
- Scoring `authoritative=True` promotion.
- Grote frontend-refactor.
- Nieuwe strategy families of crypto-diagnostic promotion als
  omweg om paper-validatie te vermijden.

## 12. Waar te beginnen voor v3.15

Roadmap-volgorde blijft leidend: **v3.15 = Paper Validation Engine**.

Concrete stappen:

1. Typed timestamped returns stream als precision-upgrade (zie
   §8.1). Bouw deze als additieve sidecar zonder de
   `candidate_returns_latest.v1.json` shape te breken.
2. Paper simulation module die `candidate_returns` en
   `portfolio_diagnostics.equal_weight_portfolio` consumeert als
   seeds. Geen live trading, geen broker — simulatie-layer
   strikt geïsoleerd.
3. Eventueel regime-filtered preset promotion als de v3.14
   evidence (diagnostics + overlay) het verantwoordt. Begin
   diagnostic, bump alleen als meerdere runs consistent zijn.
4. Scoring bump `v0.1-experimental → v0.2-experimental` met
   regime-breadth als additieve component — `authoritative=False`
   blijft, goldens updated met explicit refresh in PR description.

Leidende filosofie:
`research quality → candidate quality → regime quality →
portfolio quality → paper quality → live quality`.

v3.14 heeft **portfolio quality** geleverd. v3.15 gebruikt dat als
input, bouwt **paper quality**, maar blijft buiten live-semantiek.
