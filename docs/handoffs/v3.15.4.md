# v3.15.4 — Finalization Notes

Date: 2026-04-26
Branch: `chore/v3.15.4-finalize`
Closes the v3.15.x line; v3.16 may start.

This handoff documents two operational coupling points that the
v3.15.4 working set leaves in place. They are *intentional* under
the current single-worker assumption but become wrong the moment
that assumption changes — flag them on the v3.16 entry checklist.

## 1. Campaign launcher holds the queue lock across the subprocess

`research/campaign_launcher.py` acquires `acquire_queue_lock()`
near the top of `main()` and only releases it after the spawned
`research/run_research.py` subprocess exits, the registry / queue /
ledger / budget mutations are written, and the daily digest is
emitted. Concretely the lock spans:

1. Load all artifacts under the queue lock.
2. `decide(...)`.
3. Apply the decision (`upsert_record`, `set_lease`,
   `add_reservation`, ...).
4. **Subprocess invocation** (`_invoke_subprocess(...)`, up to
   `lease_ttl_seconds - 300` seconds; default ~2h cap).
5. Read `paper_readiness_latest.v1.json`, classify outcome,
   `record_outcome`, `transition_state`, `clear_lease`,
   `settle_reservation`.
6. Write digest, write registry / queue / ledger.
7. Release lock.

This is correct **and intentional** while
`research.campaign_templates.DEFAULT_CONFIG.max_concurrent_campaigns
== 1`. With a single worker the lock IS the worker; serialising
hourly ticks on it is exactly the desired behaviour. A second
launcher that lands while the first is mid-subprocess sees
`worker_busy → idle_noop` and exits 0.

If `max_concurrent_campaigns` is ever raised:

- the lock must be released at the *spawn* boundary (after step 3),
- step 5 must re-acquire the lock for the outcome-recording phase,
- step 5 must additionally guard against another tick having
  spawned a follow-up against a registry view that did not yet
  observe this run's `record_outcome`.

The v3.15.4 ownership stamp on `paper_readiness_latest.v1.json`
(see §2) is a prerequisite for that future work because it lets
the outcome-recording phase verify the sidecar belongs to the
campaign it is about to settle.

Until then: do **not** raise `max_concurrent_campaigns` without
this refactor. The unit test
`tests/integration/test_campaign_launcher_single_tick.py::
test_second_tick_respects_single_worker` pins the
`worker_busy → idle_noop` behaviour as the contract.

## 2. `paper_readiness_latest.v1.json` ownership stamp

Added in v3.15.4: nullable top-level field `col_campaign_id`.

- **Producer.** `research.run_research` stamps the field from the
  v3.15.2 `_COL_CAMPAIGN_ID` breadcrumb (set from the
  `--campaign-id` CLI arg). Direct CLI invocations leave the field
  `null`.
- **Consumer.** `campaign_launcher._classify_outcome_from_paper`
  takes an `expected_campaign_id` argument. When provided, the
  loaded sidecar's `col_campaign_id` MUST equal it. A missing or
  mismatched stamp means the file is either pre-v3.15.4 or
  belongs to a previous campaign whose subprocess crashed before
  overwriting it; the function returns `(None, None)` so the
  caller falls through to the conservative
  `completed_no_survivor / duplicate_low_value_run` classification
  rather than crediting / blaming the current run for another
  campaign's verdict.
- **Schema.** `PAPER_READINESS_SCHEMA_VERSION` stays at `1.0`.
  The new field is additive and nullable; pre-v3.15.4 readers
  ignore unknown keys, so dashboard / report consumers are
  unaffected.
- **Tests.**
  `tests/unit/test_paper_readiness_col_ownership.py` (5) +
  `tests/unit/test_campaign_launcher_paper_ownership.py` (7) pin
  payload shape, façade plumbing, owner-match success, the defining
  owner-mismatch rejection (stale sidecar from a prior campaign),
  pre-v3.15.4 missing-owner rejection when an owner is expected,
  the no-`expected_campaign_id` back-compat path, and corrupt /
  missing sidecar handling.

This change is a no-op for anyone who does not set
`expected_campaign_id`. The launcher is the only call site that
sets it.

## 3. Bridge-validator error readability

`validate_active_discovery_preset_bridges()` now surfaces:

- the offending `hypothesis_id`,
- its `strategy_family`,
- and (when the binding is missing) the list of presets that DO
  bind via `hypothesis_id` but failed the stable+enabled gate, with
  their actual `status` and `enabled` flags.

If no preset binds via the id at all the message says so
explicitly (`no presets bind via hypothesis_id at all`). This
distinguishes "binding is missing entirely" from "binding exists
but was flipped to `enabled=False`" — the latter is the
operationally common failure mode and was previously invisible in
the error string.

The bundle-resolves-to-zero-strategies branch already carried
preset name + bundle and now additionally surfaces
`strategy_family` for symmetry.
