## 4. Architecturele keuzes (met motivatie)

1. **Paper-validation engine als eigen laag**, niet diagnostics. Ledger + divergence + readiness zijn first-class surfaces; venue-delta is een subcomponent binnen divergence.
2. **Gesloten event-taxonomy + gesloten blocking-reason taxonomy**, versioned. Mirrort `rejection_taxonomy.py` discipline.
3. **`live_eligible=false` hard gepind** op alle v3.15 artifacts. Expliciete tests (`test_paper_no_live_invariant.py`) pinnen dit via import-scan + source-sweep + socket monkeypatch.
4. **Unknown asset_type → blocking, geen fallback**. Conservatief: geen paper-verdict zonder expliciete venue-mapping. Polymarket gedefinieerd maar niet applicable (geen Polymarket candidates in research pipeline).
5. **Venue-delta framing, niet replacement**. Engine baseline blijft referentie; v3.15 rapporteert delta t.o.v. baseline — geen dubbele cost-charge, geen engine-wijziging. `cost_sensitivity` math hergebruikt, niet gedupliceerd.
6. **Byte-identity op controlled fixtures**, niet pipeline output. v3.15 regression tests lopen tegen fixtures, niet tegen live pipeline runs.
7. **Geen broker/live calls**. Module-level import-invariant getest via dedicated scan.
8. **Shared `_normalize_stream` helper**. Geen gedragswijziging op `portfolio_reporting`; extracted voor hergebruik.
9. **Runner blijft dun**. Eén additieve block na de v3.14 hook; alle logica in de façade.
10. **Scoring blijft v0.1-experimental**. Geen bump in v3.15.

## 8. Bekende beperkingen (voor v3.16 aan te pakken)

### 8.1 Bar-level paper-returns stream deferred

v3.15 gebruikt het analytische per-fill multiplicatieve model (scalar-equivalent aan `cost_sensitivity`). Bar-exact replay — waarbij we per bar een venue-adjusted return krijgen — vereist `build_cost_sensitivity_report` aan te roepen met de volledige `oos_bar_returns` + `fill_positions` plumbing. Voor v3.16:

- Plumb `oos_bar_returns` + `oos_execution_events` + `baseline_dag_returns` + `kosten_per_kant` per evaluation door naar de façade
- Bouw `build_cost_sensitivity_report` calls per candidate met `[baseline_scenario, venue_scenario]`
- Emit een nieuwe additieve sidecar `research/paper_timestamped_returns_latest.v1.json` met bar-level venue-adjusted returns, zodat `paper_divergence.timestamp_aligned_return_diff` een echte per-bar diff kan rapporteren i.p.v. alleen de coverage window

### 8.2 Polymarket venue gedefinieerd, niet toegepast

Wacht op Polymarket candidates via Bot / DataArbitrage agent research-integratie. De constants (2% spread, 10 bps) zijn gedefinieerd; `venue_name_for_asset_type("polymarket")` returnt nu `None`. Wanneer Polymarket candidates de research pipeline bereiken kan de mapping activeren via toevoeging van `"polymarket"` als normalizable asset_type in `research/asset_typing.py`.

### 8.3 Scoring blijft v0.1-experimental

Bewust niet gebumped in v3.15. v3.16 kan `SCORING_FORMULA_VERSION` naar `v0.2-experimental` bumpen (authoritative blijft `False`) met `regime_breadth_signal` als additieve component — alleen wanneer meerdere runs met consistent bewijs het rechtvaardigen.

### 8.4 Negative paper sharpe escalatie

In v3.15 is `negative_paper_sharpe` een **warning**, niet blocking. De escalatie-regel (promoten tot blocking bij samenval met `excessive_divergence` of `no_candidate_returns` / `missing_execution_events`) is ontworpen maar in code niet actief ingevoerd omdat de bestaande blockers de overlap al coveren. v3.16 kan expliciet een `negative_paper_sharpe_escalated` blocking-code toevoegen indien operationele ervaring aantoont dat de overlap niet voldoet.

### 8.5 Geen paper-to-live promotion

`live_eligible=False` is hard gepind. v3.16 kan — na meerdere runs met consistent `ready_for_paper_promotion` voor kandidaten — een aparte **paper-promotion layer** bouwen die eerst kandidaten van "research" naar "paper" levensfase promoveert (zonder live). Een latere v3.17+ kan dan paper-to-live overwegen, mits strict gated.

### 8.6 Geen frontend surface

Consumptie gaat via 4 endpoints + markdown report. v3.16+ kan een read-only paper-tab toevoegen (lezing van `GET /api/registry/paper` summary) wanneer operationele behoefte bevestigd is.

## 12. Waar te beginnen voor v3.16

Roadmap-volgorde blijft leidend:
`research → candidate → regime → portfolio → paper → live`

v3.15 heeft **paper quality** geleverd als volwaardige validation
engine met ledger + divergence + readiness. v3.16 bouwt
bar-level precision op die laag en begint eventueel aan paper-
promotion — nog steeds research-only.

Concrete stappen:

1. Bar-level paper-returns stream (§8.1) als additieve sidecar.
2. Scoring bump `v0.1-experimental → v0.2-experimental` met
   `regime_breadth_signal` (additieve component,
   `authoritative=False` blijft).
3. Polymarket venue activation wanneer pipeline-candidates
   beschikbaar zijn.
4. Paper-promotion layer — van `candidate` naar
   `paper_ready` lifecycle, gestuurd door
   `paper_readiness.readiness_status=="ready_for_paper_promotion"`
   over meerdere runs.

Leidende filosofie blijft onveranderd: diagnostic-first, named
thresholds, gesloten taxonomies, additieve sidecars, geen
engine-contract uitbreiding, geen live surfaces.
