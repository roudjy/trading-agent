# v3.15.10 — Funnel Completion (combined: v3.15.8 + v3.15.9 + v3.15.10)

Date: 2026-04-26
Branch: `feature/v3.15.8-15.10-funnel-completion`

Single VERSION bump. No intermediate `3.15.8` / `3.15.9`
releases were tagged; their work ships as part of this combined
release.

---

## 1. Why combined

Three layers of the funnel were incomplete after v3.15.7:

1. **Sampling** — the `screening_param_samples` shim evaluated at
   most 3 combinations (first / middle / last). For the small
   exploratory grids introduced in v3.15.6 that meant coverage was
   often <40%, and exploratory passes were structurally
   under-counted.
2. **Evidence** — there was no per-candidate artifact explaining
   *why* a candidate passed, failed, or was a near-miss. v3.15.7
   `pass_kind` and `diagnostic_metrics` lived only inside the
   internal screening sidecar.
3. **Policy** — the campaign system had no consumer of funnel
   evidence: exploratory passes downgraded to
   `needs_investigation` produced no follow-up campaign;
   near-passes were invisible; low coverage was invisible;
   `technical_failure` shared cooldown logic with
   `research_rejection`.

This release closes those three layers and leaves the funnel
end-to-end traceable:

```
strategy → exploratory screening → shortlist / needs_investigation
       → confirmation request → promotion → paper
```

No threshold changes, no new strategies, no taxonomy changes,
no frozen-contract mutation.

## 2. v3.15.8 — Parameter Sampling Calibration

### 2.1 Behaviour table

| `param_grid` | `grid_size` | samples returned | `sampling_policy` | `coverage_pct` | `coverage_warning` |
|---|---|---|---|---|---|
| `None` / not a dict | `None` | `[{}]` | `grid_size_unavailable` | `None` | `grid_size_unavailable` |
| Expansion raises (malformed) | `None` | `[{}]` | `grid_size_unavailable` | `None` | `grid_size_unavailable` |
| All-empty axes | `None` | `[{}]` | `grid_size_unavailable` | `None` | `grid_size_unavailable` |
| `{}` | 1 | `[{}]` | `empty_grid` | 1.0 | `None` |
| 1..8 | N | all N | `full_coverage` | 1.0 | `None` |
| 9..16 | N | K (≥80%, endpoints included) | `stratified_small` | K/N | `None` |
| > 16 | N | first/middle/last (3) | `legacy_large_grid` | 3/N | `None` |

Constants live in
[research/candidate_pipeline.py](research/candidate_pipeline.py):
`MAX_FULL_COVERAGE_GRID_SIZE = 8`,
`MAX_STRATIFIED_GRID_SIZE = 16`,
`MIN_STRATIFIED_COVERAGE_PCT = 0.80`,
`LEGACY_LARGE_GRID_SAMPLE_COUNT = 3`.

### 2.2 Behavioural shift (intentional, documented)

The legacy `screening_param_samples(param_grid, max_samples=N)`
shim now ignores `max_samples` for grid_size in
`[1..MAX_STRATIFIED_GRID_SIZE]`. This is the deliberate fix for
the v3.15.7 under-sampling defect: small exploratory grids were
evaluated with at most 3 combinations even when the full grid
was tractable.

For grid_size > 16 the legacy `max_samples` cap is still honoured
so callers that explicitly cap the legacy regime continue to
work.

Pinned by
[tests/unit/test_v3_15_8_legacy_shim_intentionally_ignores_max_samples_for_small_grids.py](tests/unit/test_v3_15_8_legacy_shim_intentionally_ignores_max_samples_for_small_grids.py).

### 2.3 Budget impact

`MAX_STRATIFIED_GRID_SIZE = 16` is the per-candidate hard
evaluation cap. The existing
`DEFAULT_SCREENING_CANDIDATE_BUDGET_SECONDS = 300` wall deadline
plus the engine-level early-exits at
[research/screening_runtime.py:245](research/screening_runtime.py:245)
and [:306](research/screening_runtime.py:306) prevent runaway:
the loop short-circuits as soon as the deadline is reached.

### 2.4 Coverage metadata in the outcome dict

Every screening outcome dict now carries a `sampling` block on
**every** return path (success, error, timeout, no-engine fast
path). When the caller does not supply explicit metadata the
fallback `_UNAVAILABLE_SAMPLING_METADATA` shape is used so
downstream evidence and policy never see missing keys.

`sampled_parameter_digest` is a sha1 hex of the canonical-JSON
sample list with `allow_nan=False` — NaN/inf parameter values
must be sanitised upstream by `_json_safe_param_value` (which
collapses them to `None`).

## 3. v3.15.9 — Funnel Evidence Artifact

### 3.1 New non-frozen sidecar

`research/screening_evidence_latest.v1.json` — `schema_version`
`"1.0"`. Written by
[research/run_research.py](research/run_research.py) adjacent to
the v3.15.3 catalog/metadata block, AFTER paper validation so
paper_blocked signals can fold in.

### 3.2 Top-level schema (closed; pinned)

```json
{
  "schema_version": "1.0",
  "generated_at_utc": "...",
  "git_revision": "...",
  "run_id": "...",
  "campaign_id": "... or null",
  "col_campaign_id": "... or null",
  "preset_name": "... or null",
  "screening_phase": "exploratory|standard|promotion_grade|null",
  "artifact_fingerprint": "<sha1 hex>",
  "summary": {...},
  "candidates": [...]
}
```

### 3.3 Per-candidate schema (closed; pinned)

Includes `evidence_fingerprint` (used by v3.15.10 dedupe) and
`identity_fallback_used` (operator signal — see §3.5).

### 3.4 Ownership + fingerprint design (MF-7, MF-8)

`col_campaign_id` is the authoritative owner; `campaign_id` is a
redundant alias in this release. v3.15.10 ownership lookup
prefers `col_campaign_id`, falling back to `campaign_id`.

Fingerprints are sha1 of canonical JSON with
`allow_nan=False`. NaN/inf metric values are coerced to `None`
upstream by `to_json_safe_float` so the artifact never carries
unsanitised non-finite floats.

### 3.5 Identity-fallback resilience (MF-6, MF-19)

The builder NEVER asserts on a malformed candidate dict. A
missing/empty/whitespace `candidate_id` triggers a deterministic
`fb_<sha1prefix>` fallback id and the per-candidate record sets
`identity_fallback_used: true`. The run-level summary
increments `identity_fallbacks` so operators can spot real
upstream defects.

### 3.6 Stage-result two-step resolution (MF-5)

```
Step 1 (base from screening promotion):
  promoted is None      -> unknown
  promoted is False:
      near-pass true    -> near_pass
      near-pass false   -> screening_reject
  promoted is True      -> screening_pass

Step 2 (downstream override only on a screening pass):
  paper_blocked         -> paper_blocked
  pass_kind == exploratory OR
    promotion_status == needs_investigation
                        -> needs_investigation
  pass_kind in {standard, promotion_grade}
                        -> promotion_candidate
  else                  -> screening_pass
```

A rejected near-pass remains `near_pass`; downstream
`paper_blocked` / `promotion_candidate` / `needs_investigation`
cannot override a rejection. `unknown` cannot silently flip to
`paper_blocked`.

### 3.7 Near-pass logic (REV 3 §6.10)

Conservative classifier — fires only when:

- `screening_phase == "exploratory"`,
- exactly ONE failure reason present,
- that reason is in `NEAR_PASS_ELIGIBLE_REASONS`,
- the relevant metric lies within its near-band:
  - `expectancy_not_positive`: `-0.0005 <= expectancy <= 0`
  - `profit_factor_below_floor`: `pf >= 1.05 * 0.95`
  - `drawdown_above_exploratory_limit`: `dd <= 0.45 * 1.05`

`insufficient_trades`, `no_oos_samples`,
`candidate_budget_exceeded`, `screening_candidate_error` are
NEVER near-pass. Near-pass NEVER mutates pass/fail —
invariant test in
[tests/unit/test_v3_15_9_stage_result_resolution.py](tests/unit/test_v3_15_9_stage_result_resolution.py).

### 3.8 Paper-blocked graceful degradation (MF-12)

`_read_paper_blocked_index()` returns `{}` when
`research/paper_readiness_latest.v1.json` is absent or
malformed. The evidence artifact still writes; downstream
stage_result falls back to the screening/promotion state.

## 4. v3.15.10 — Campaign Policy Alignment

### 4.1 Decision codes + priorities

| Decision code | Priority | Spawn? |
|---|---|---|
| `confirmation_from_exploratory_pass` | 10 | YES (decision-only metadata, see §4.4) |
| `follow_up_from_near_pass` | 20 | YES |
| `alternate_timeframe_from_insufficient_trades` | 30 | YES (when supported — never in v3.15.10) |
| `coverage_followup_from_low_sampling_coverage` | 40 | NO |
| `cooldown_from_repeat_rejection` | 50 | NO |
| `no_action_technical_failure` | 60 | NO |

Sort key: `(priority, decision_code, preset_name, strategy_id, candidate_id)`.

### 4.2 Decision rules (REV 3 §7.3)

| Per-candidate evidence input | Decision |
|---|---|
| `stage_result == "needs_investigation"` (exploratory pass) | `confirmation_from_exploratory_pass` |
| `stage_result == "near_pass"` AND `near_pass.is_near_pass == True` | `follow_up_from_near_pass` |
| `failure_reasons == ["insufficient_trades"]` | `cooldown_from_repeat_rejection` (rationale `alternate_timeframe_unavailable=True`) |
| `coverage_warning ∈ {below_threshold_for_small_grid, grid_size_unavailable}` OR `sampling_policy == "grid_size_unavailable"` | `coverage_followup_from_low_sampling_coverage` |

| Per-campaign / preset-level | Decision |
|---|---|
| Registry record `outcome == "technical_failure"` | `no_action_technical_failure` (DOES NOT require evidence) |
| Repeat-rejection streak ≥ 3 with same `dominant_reason` | `cooldown_from_repeat_rejection` |

`degenerate_no_survivors` retains its existing
meaningful-signal classification — funnel emits no decisions
for degenerate parents.

### 4.3 Repeat rejection streak — exact (REV 3 §7.6)

Walks the tail of `campaign_completed` events for the preset.
Increments when `outcome == "research_rejection"` AND the
event-level dominant reason matches. `technical_failure` and
`degenerate_no_survivors` are NEUTRAL (skip without breaking).
Any other outcome BREAKS the streak. Threshold is 3.

### 4.4 LIMITATION — `requested_screening_phase` is decision-only

There is currently no executor reader of
`extra.requested_screening_phase` anywhere in the `research/`
subtree (pinned by static-grep
[tests/unit/test_v3_15_10_safety_invariants.py](tests/unit/test_v3_15_10_safety_invariants.py)).

The funnel-policy decision for `confirmation_from_exploratory_pass`
populates a `survivor_confirmation` `SpawnRequest` with
`extra.requested_screening_phase = "promotion_grade"`. v3.15.10
records the request as **metadata only** — the spawned campaign
runs at the parent preset's natural `screening_phase` (i.e.
`exploratory`), NOT at `promotion_grade`. v3.15.11+ may wire
executor support for the requested-phase override; until then
the metadata is forensic / documentation-only.

Vocabulary: throughout this release, the action is described as
a **"confirmation request"** or **"queued confirmation
decision"** — not "promotion-grade confirmation execution".

### 4.5 LIMITATION — alternate-timeframe execution

There is no preset-catalog mechanism for "same hypothesis,
different timeframe" today. `has_alternate_timeframe_support()`
explicitly returns `False`. `insufficient_trades` failures
therefore always emit
`cooldown_from_repeat_rejection` with rationale
`alternate_timeframe_unavailable=True`. Lifting this requires a
preset/hypothesis catalog extension (out of scope for v3.15.10).

### 4.6 Launcher integration — error-isolated (REV 3 §7.11 + MF-9)

`research/campaign_launcher.py` calls a new
`_apply_funnel_decisions(...)` helper AFTER the existing
`_apply_decision(...)` and BEFORE `_write_ledger(...)` in the
per-tick critical section.

The whole block is wrapped in `try/except`. Failures emit a
single `funnel_policy_error` event but NEVER mutate the parent
campaign's outcome. Per-decision failures emit a per-decision
`funnel_policy_error`.

### 4.7 v3.15.10 emits LEDGER EVENTS ONLY

v3.15.10 deliberately does NOT mutate the registry or queue
from the funnel hook. Decisions are recorded as ledger events
and the `spawn_request` payload is carried in the event's
`extra` so a future v3.15.11+ executor can act on it. This
matches the "decision-only metadata" framing of MF-15 and keeps
existing followup mechanics undisturbed.

### 4.8 Dedupe — exact (REV 3 §7.9 + MF-15)

Ledger-based dedupe via
`_funnel_decision_already_in_ledger(events, parent_campaign_id,
decision_code, candidate_id, evidence_fingerprint)`. A second
tick with a prior `funnel_decision_emitted` event matching the
same tuple is skipped.

The pure module also exports `has_funnel_spawn_for(...)` for
registry-based dedupe (reserved for v3.15.11+ when actual
campaign records are spawned). Terminal/active state semantics
are pinned against the real `CAMPAIGN_STATES` tuple in
[research/campaign_registry.py](research/campaign_registry.py).

### 4.9 Ledger — additive event types (REV 3 §6.12)

Four new entries in the closed `EventType` Literal at
[research/campaign_evidence_ledger.py](research/campaign_evidence_ledger.py):

- `funnel_decision_emitted`
- `funnel_evidence_stale_or_mismatched`
- `funnel_technical_no_freeze`
- `funnel_policy_error`

`LEDGER_SCHEMA_VERSION` stays at `"1.0"` — the on-disk JSONL
row schema is unchanged.

### 4.10 Digest — schema 1.0 → 1.1 with backward read (REV 3 §6.13 + MF-11)

Two readers exist:
- [research/campaign_launcher.py:353](research/campaign_launcher.py:353)
  via `load_previous_digest()`.
- [dashboard/api_campaigns.py:90](dashboard/api_campaigns.py:90)
  via `_read_json(DIGEST_ARTIFACT_PATH, {})`.

Both consume the dict via `.get(...)` style access; adding the
new top-level `funnel_decisions: dict[decision_code -> count]`
key does not break either. v1.0 fixture loadability is pinned
by
[tests/regression/test_v3_15_10_digest_v1_0_legacy_loadable.py](tests/regression/test_v3_15_10_digest_v1_0_legacy_loadable.py).

## 5. How earlier versions are consumed

### v3.15.5 outcome semantics

The funnel-policy `repeat_rejection_streak()` skips
`technical_failure` and `degenerate_no_survivors` outcomes
(both NEUTRAL — neither increment nor break the streak). The
v3.15.5 `worker_crashed`-never-emitted invariant is preserved.

### v3.15.6 `screening_phase`

Per-candidate evidence record carries `screening_phase` from
`preset_obj.screening_phase`. The evidence builder uses it to
dispatch near-pass eligibility (only `exploratory` qualifies)
and the criteria split (exploratory hard gates vs legacy
AND-gate).

### v3.15.7 exploratory criteria + `pass_kind`

The screening outcome dict's v3.15.7 `pass_kind` field is
threaded into the evidence record. `pass_kind == "exploratory"`
maps to `stage_result == "needs_investigation"` per
v3.15.7's `classify_candidate` short-circuit, which v3.15.10
funnel policy maps to `confirmation_from_exploratory_pass`.

## 7. Operator guide — reading the evidence artifact

```bash
jq '.schema_version, .run_id, .campaign_id, .col_campaign_id, .artifact_fingerprint' \
   research/screening_evidence_latest.v1.json

jq '.summary' research/screening_evidence_latest.v1.json

                  | {candidate_id, strategy_name, asset, interval,
                     metrics, evidence_fingerprint}' \
   research/screening_evidence_latest.v1.json

                  | {candidate_id, near_pass, metrics}' \
   research/screening_evidence_latest.v1.json
```

```bash
   research/campaign_evidence_ledger_latest.v1.jsonl | tail -20
```

## 8. Caution — diagnosing zero exploratory passes

If post-deploy the exploratory pass count remains zero, inspect
in this order **before** changing any threshold:

1. **Sampling coverage** — `summary.coverage_warnings > 0` in
   the evidence artifact suggests `grid_size_unavailable` or
   `below_threshold_for_small_grid` regimes are in play.
   Investigate the offending strategy's `params` upstream.
2. **Evidence failure reasons** —
   `summary.dominant_failure_reasons[0]` tells you which
   exploratory criterion is filtering out candidates.
3. **Policy decisions** — the digest's `funnel_decisions` block
   shows whether confirmation requests are being recorded;
   `funnel_evidence_stale_or_mismatched` events indicate the
   evidence artifact is out of sync with the campaign launcher's
   active campaign id.

Only after these three checks come back inconclusive should
v3.15.7 thresholds be revisited.
