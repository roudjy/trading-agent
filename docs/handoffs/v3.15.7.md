## 1. Funnel rationale

```
Screening zoekt.        ← v3.15.7 herstelt deze laag voor exploratory.
Promotion bewijst.      ← byte-identiek aan pre-v3.15.7.
Paper valideert.        ← byte-identiek aan pre-v3.15.7.
```

Pre-v3.15.7 was screening impliciet altijd promotion-grade.
Trend/momentum-strategieën hebben structureel `win_rate < 0.50`
(weinig wins maar groot; veel verliezen maar klein) en faalden
daardoor permanent in screening — terwijl ze bewezen positieve
expectancy hebben. v3.15.7 maakt **exploratory** discovery-
compatible zonder de eindlat te verlagen: een exploratory pass
is een **shortlist signal**, geen promotion candidate.

## 2. Criteria per phase

| Phase | Criteria-fn | Hard gates | Diagnostic-only | Reason on fail |
|---|---|---|---|---|
| `promotion_grade` | `_legacy_criteria` | engine `goedgekeurd` (5-criteria AND-gate) | — | `screening_criteria_not_met` |
| `standard` | `_legacy_criteria` (conservative) | idem | — | `screening_criteria_not_met` |
| `None` (back-compat) | `_legacy_criteria` | idem | — | `screening_criteria_not_met` |
| `exploratory` | `_exploratory_criteria` | `expectancy > 0`; `profit_factor >= 1.05`; `max_drawdown <= 0.45`; trade-count via existing `engine.min_trades` pre-check upstream | `win_rate`, `sharpe`, `sortino`, `consistentie` | `expectancy_not_positive` / `profit_factor_below_floor` / `drawdown_above_exploratory_limit` / `insufficient_trades` |

Threshold values are **start-points**; v3.15.8 (sampling
calibration) may recalibrate after observing real exploratory
pass count. See §10.

## 3. Why win_rate is diagnostic-only for exploratory

Trend / momentum strategies systematically produce
`win_rate < 0.50` because they win infrequently but with large
gains and lose often with small bounded losses. The promotion-
grade `win_rate > 0.50` AND-gate filters them out at screening
*before* their expectancy can be measured downstream. v3.15.7's
exploratory phase replaces the win_rate gate with a
**scale-weighted edge proxy** (expectancy + profit_factor):
- `expectancy > 0` says the average trade is net-positive.
- `profit_factor >= 1.05` says total wins outweigh total losses
  with a small margin floor.

`win_rate` is still surfaced in `diagnostic_metrics` for
operators, but it doesn't trigger a hard fail.

## 4. Metrics definitions

### expectancy

```
expectancy = mean(trade_pnls)
```

Engine `trade_pnls` are per-trade fractional returns net of costs
(0.05 = +5%; not notional EUR — see
[agent/backtesting/engine.py:1409](agent/backtesting/engine.py:1409)).
Edge cases: empty trade list → `0.0`.

### profit_factor

```
gross_profit = sum(p for p in trade_pnls if p > 0)
gross_loss   = abs(sum(p for p in trade_pnls if p < 0))

if gross_loss > 0:        profit_factor = gross_profit / gross_loss
elif gross_profit > 0:    profit_factor = PROFIT_FACTOR_NO_LOSS_CAP   # = 999.0
else:                     profit_factor = 0.0
```

`PROFIT_FACTOR_NO_LOSS_CAP = 999.0` is a deterministic JSON-safe
finite proxy for "no losses observed" — `float("inf")` would break
`json.dumps(allow_nan=False)`. Edge cases: empty / all-losers →
`0.0`; all-winners → `999.0`; mixed → ratio.

### max_drawdown

Engine output is **always positive**, scale `0..1`
(`abs(dd.min())` on the daily-equity path). v3.15.7's exploratory
threshold `<= 0.45` operates on this same scale.

---

## 5. profit_factor cap rationale

Without a cap, an "all-winners" sample produces `inf`, which:
- breaks `json.dumps(allow_nan=False)` at sidecar write time
- complicates downstream comparisons / logging

`999.0` is the chosen finite proxy:
- monotone with the natural ratio (always larger than any real
  observed mixed ratio in finite trade counts)
- recognisable by ops as "cap hit, no losses observed"
- safe under all JSON serialisers

## 7. SCREENING_PROMOTED semantic disambiguation

Historically `SCREENING_PROMOTED` (= `"promoted_to_validation"`)
meant "this candidate passed screening and proceeds to the next
pipeline stage" — **NOT** "this is a promotion candidate". v3.15.7
disambiguates this via `pass_kind`:

| Screening status | `pass_kind` | Semantic meaning |
|---|---|---|
| `SCREENING_PROMOTED` | `None / standard / promotion_grade` | Promotion candidate (pre-v3.15.7 behavior) |
| `SCREENING_PROMOTED` | `exploratory` | **Exploratory shortlist signal** — needs investigation, not promotion |
| `SCREENING_REJECTED` | `None` | Rejected (failure_reason carries semantics) |

The constant name stays `SCREENING_PROMOTED` to avoid touching
[research/candidate_pipeline.py](research/candidate_pipeline.py) and the
many existing call sites. Future versions may rename to
`SCREENING_PASSED` if a broader cleanup happens; for now the
disambiguation lives in `pass_kind`.

## 8. Failure reason taxonomy

`SCREENING_REASON_CODES` v3.15.7 (extends v3.15.5 base):

```python
SCREENING_REASON_CODES = frozenset({
    # v3.15.5
    "insufficient_trades",
    "no_oos_samples",
    "screening_criteria_not_met",
    # v3.15.7
    "expectancy_not_positive",
    "profit_factor_below_floor",
    "drawdown_above_exploratory_limit",
})
```

The three new codes are emitted exclusively by
`research/screening_criteria.py::_exploratory_criteria` via the
phase-aware dispatch in `research/screening_runtime.py`. Including
them in `SCREENING_REASON_CODES` keeps v3.15.5
`_classify_research_rejection` correctly classifying a fully-
exploratory-rejected run as `research_rejection`.

A defensive test (`test_v3_15_7_screening_reason_codes_extended`)
greps every emitted reason literal and verifies it lives in the
taxonomy — drift is caught at CI time.

## 10. v3.15.8 sampling dependency

If after v3.15.7 deploy the **exploratory pass count remains zero**
across multiple runs, **inspect sampling coverage in v3.15.8 first
before further loosening criteria**. Reason: v3.15.7's criteria
change without a sampling-coverage fix may mask a sampling
bottleneck — the exploratory funnel can't pass strategies it
never samples densely enough to evaluate.

Indicators that v3.15.8 sampling work is needed:

- exploratory pass count == 0 across all v3.15.6 exploratory
  presets (`trend_pullback_crypto_1h`,
  `vol_compression_breakout_crypto_1h`, `crypto_diagnostic_1h`)
- `samples_total` per candidate consistently low
- `expectancy_not_positive` or `profit_factor_below_floor`
  appears at the same rate as pre-v3.15.7's
  `screening_criteria_not_met` — suggesting the strategies
  simply aren't getting enough parameter coverage to surface
  a positive edge

Do NOT respond to such signals by lowering exploratory thresholds
without first investigating sampling. The v3.15.7 thresholds
(`expectancy > 0`, `profit_factor >= 1.05`, `max_drawdown
<= 0.45`) are intentionally minimal — lowering them further would
admit clearly-not-edge candidates.
