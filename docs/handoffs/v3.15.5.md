# v3.15.5 — Outcome Semantics Fix

Date: 2026-04-26
Branch: `fix/v3.15.5-outcome-semantics`
Closes v3.15.3 carry-over **C1**.

This handoff documents the post-v3.15.5 launcher emission contract,
the ownership chain that makes the new ``research_rejection``
classification deterministic, and the behavioral shift downstream
consumers need to be aware of.

---

## 1. Problem solved

Pre-v3.15.5, every subprocess that exited non-zero — including a
controlled `DegenerateResearchRunError` raise from
[research/run_research.py:1835](../../research/run_research.py:1835)
— landed in the launcher's single
`outcome = "worker_crashed"` default at
[research/campaign_launcher.py:796](../../research/campaign_launcher.py:796).

Side effects:

- `_non_technical_reject_streak` excluded these runs (technical
  reason code) → preset freeze never fired on degenerate streaks.
- `meaningful_classification = "uninformative_technical_failure"`
  poisoned digest counts.
- `top_failure_reasons` reported `worker_crash` where the truth was
  `degenerate_no_evaluable_pairs`.
- Roadmap v3 §9 (v3.15.2 Autonomous Campaign Operations Layer)
  reads outcome / meaningful_classification as policy input;
  building it on a poisoned signal would have been malpractice.

This is a **semantics bug**, not a runtime bug. The fix is
narrow: correct the outcome mapping, leave everything else intact.

---

## 2. Post-v3.15.5 launcher emission contract

The launcher MUST emit exactly one of the values in
`research.campaign_registry.LAUNCHER_EMITTABLE_OUTCOMES`. The
runtime invariant in `_apply_decision` asserts this directly. The
deprecated `worker_crashed` literal stays in
`CampaignOutcome` / `CAMPAIGN_OUTCOMES` only for reading legacy
records — the launcher must never produce it post-v3.15.5.

### Mapping (deterministic, exhaustive)

```
rc == EXIT_CODE_DEGENERATE_NO_SURVIVORS (= 2)
  → outcome="degenerate_no_survivors"
    meaningful="meaningful_failure_confirmed"
    reason_code="degenerate_no_evaluable_pairs"

rc != 0 and rc != 2
  → outcome="technical_failure"
    meaningful="uninformative_technical_failure"
    reason_code = "timeout" if rc==124 else "worker_crash"

rc == 0:
  Stage A: paper_readiness.status == "ready_for_paper_promotion"
           AND col_campaign_id == expected_campaign_id
    → outcome="completed_with_candidates" (existing path)
  Stage B: paper_readiness.status == "blocked"
           AND col_campaign_id == expected_campaign_id
    → outcome="paper_blocked" (existing path)
  Stage C: paper_readiness col_campaign_id == expected_campaign_id
           AND paper_readiness.status NOT in (ready, blocked)
           AND candidate_registry_v1 readable
           AND len(candidates) > 0
           AND all candidates.status == "rejected"
           AND failed-reason-union ⊆ SCREENING_REASON_CODES
           AND failed-reason-union not empty
    → outcome="research_rejection"
      meaningful="meaningful_family_falsified"
      reason_code = dominant screening reason (alphabetical tiebreak)
  Stage D: residual fallback (everything else)
    → outcome="completed_no_survivor"
      meaningful="duplicate_low_value_run"
      reason_code="none"
```

### Lifecycle state

`degenerate_no_survivors` is a structured completion (the run did
finish, with a meaningful verdict), so its lifecycle state is
`"completed"` and the terminal event is `campaign_completed`.
`technical_failure` keeps lifecycle `"failed"` /
`campaign_failed`. This makes the existing
`_non_technical_reject_streak` counter — which inspects only
`campaign_completed` events — observe degenerate runs correctly.

## 3. Ownership chain (no frozen contract mutation)

```
expected_campaign_id      (launcher: from lease)
        |
        v
paper_readiness_latest.v1.json
   .col_campaign_id    --- HARD MATCH? (single anchor)
        |
        | producer atomicity (same subprocess writes both)
        v
candidate_registry_latest.v1.json
   (frozen v1 schema; no run_id field; identity anchored above)
```

The v3.15.4 paper readiness ownership stamp does double duty in
v3.15.5: it not only protects the existing
`completed_with_candidates` / `paper_blocked` paths from stale
sidecar misclassification, it also serves as the ownership anchor
for `research_rejection`. Because `run_research` writes both
`paper_readiness_latest.v1.json` and
`candidate_registry_latest.v1.json` from the same subprocess, an
ownership match on paper readiness is sufficient to trust the
candidate registry contents for that run.

`mtime` is NOT used as ownership. It is fragile under parallel
workers, NFS, and container clock skew. The ownership rule is
strict: paper readiness ownership match or no `research_rejection`
classification — fall back to `completed_no_survivor`.

## 4. EXIT_CODE_DEGENERATE_NO_SURVIVORS = 2

Defined in
[research/empty_run_reporting.py](../../research/empty_run_reporting.py)
alongside `DegenerateResearchRunError` so the outcome semantics
live next to the exception class.

Producer: only the `__main__` wrapper of
[research/run_research.py](../../research/run_research.py:3413).
Other exceptions still propagate (Python default rc=1) →
`technical_failure`. The callable `run_research(...)` is
unchanged; library callers and the existing test
`tests/unit/test_run_research_empty_run_handling.py` continue to
see `DegenerateResearchRunError` raised.

### rc=2 launcher guard

When the launcher sees `rc == 2` it reads
`empty_run_diagnostics_latest.v1.json` (already produced by
`run_research` *before* the raise) and emits a stderr
diagnostic line:

- `rc2_origin_confirmed_degenerate` — payload present, well-formed,
  ownership matches.
- `rc2_unexpected_origin` — diagnostics file missing / ownership
  mismatch. Outcome stays `degenerate_no_survivors` (rc=2 is
  producer-restricted), but the operator gets a forensic signal.
- `rc2_payload_malformed` — payload exists but failure_stage or
  summary is missing; same outcome assignment, different log line.

The diagnostics sidecar gained a nullable top-level
`col_campaign_id` field (additive; schema_version unchanged at
`"1.0"`) so the rc=2 ownership check can run without a frozen
contract change.

## 5. Behavioral shift consumers must know about

### 5.1 Reading historical records

Pre-v3.15.5 records carrying `outcome == "worker_crashed"` describe
runs that today would be classified as either `degenerate_no_survivors`
or `technical_failure`. Without re-running them we cannot tell
which. **Recommended consumer behavior:** treat
`outcome == "worker_crashed"` as `technical_failure` for migration
windows. Treat the union
`("worker_crashed", "technical_failure")` as the canonical
"technical" filter.

We do NOT rewrite historical records. The deprecated literal stays
valid in the schema for read compatibility.

### 5.2 Policy throughput may dip initially

Pre-v3.15.5 the freeze-counter excluded degenerate runs (technical
reason code) so a preset emitting repeated degenerate runs never
froze. Post-v3.15.5 it does. Five consecutive
`degenerate_no_survivors` (or `research_rejection`) events on one
preset triggers `policy_state="frozen"` per existing rules. This is
intended — repeated degenerate runs are a structured family
falsification signal that should reduce throughput on that preset.

This may produce an initial dip in campaign throughput right after
deployment as the policy layer "catches up" to the corrected
signal. Operators should expect this.

### 5.3 Digest / dashboards

`top_failure_reasons` and `meaningful_classification` aggregations
will surface the new vocabulary (`degenerate_no_evaluable_pairs`,
`meaningful_family_falsified`) where pre-v3.15.5 they surfaced
`worker_crash` / `uninformative_technical_failure`. UI consumers
hardcoded on the legacy strings should accept both during the
migration window.
