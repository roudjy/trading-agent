# v3.15.6 — Screening Mode Activation

Date: 2026-04-26
Branch: `fix/v3.15.6-screening-mode-activation`

This release activates the funnel-stage classification
(``screening_phase``) end-to-end as plumbing only. No threshold
or screening-criteria change ships in v3.15.6 — the seam is
laid for v3.15.7 to dispatch criteria on it.

---

## 1. Summary

| Concept | Values | Meaning | Runtime effect in v3.15.6 |
|---|---|---|---|
| `screening_mode` (legacy, v3.10) | `strict` / `lenient` / `diagnostic` | gate-strictness metadata | unchanged; runtime-inert |
| `screening_phase` (NEW, v3.15.6) | `exploratory` / `standard` / `promotion_grade` | funnel-stage metadata | propagated to screening boundary; **no behavior change** |

Both fields coexist intentionally. v3.15.6 does NOT rename, retire,
or change the vocabulary of the legacy ``screening_mode``. v3.15.7
or a later cleanup phase may decide if/when to retire the legacy
field.

---

## 2. Why a new field instead of reusing ``screening_mode``

Pre-v3.15.6 ``screening_mode`` is preset-level metadata exposed via
the dashboard API ([research/presets.py:625](research/presets.py:625) `preset_to_card`),
the CLI ([researchctl.py:73](researchctl.py:73)), the run_meta sidecar
([research/run_meta.py:154](research/run_meta.py:154)), and the frontend
([frontend/src/routes/Presets.tsx:87](frontend/src/routes/Presets.tsx:87)).
Renaming the field or replacing its vocabulary would have required
frontend changes — the v3.15.6 spec forbids that.

The new ``screening_phase`` describes a fundamentally different
concept: which funnel stage a preset operates in (discovery /
default research / promotion-grade gating). The two concepts are
orthogonal:

- A `promotion_grade` preset can have any `screening_mode` (today
  all production presets use `strict`).
- An `exploratory` preset can be either `strict` or `diagnostic`
  in screening_mode.

v3.15.7 will use ``screening_phase`` (not the legacy mode) to
decide phase-aware criteria.

## 3. Preset assignments (§6 of the plan)

All 6 production presets receive an explicit ``screening_phase``
keyword in [research/presets.py](research/presets.py):

| Preset | screening_mode (legacy) | screening_phase (NEW) | Note |
|---|---|---|---|
| `trend_equities_4h_baseline` | strict | `promotion_grade` | baseline preset |
| `pairs_equities_daily_baseline` | strict | `promotion_grade` | disabled/planned, promotion-targeted on enable |
| `trend_regime_filtered_equities_4h` | strict | `promotion_grade` | diagnostic comparison; not excluded from promotion |
| `trend_pullback_crypto_1h` | strict | `exploratory` | active_discovery hypothesis. **Classification change only — same screening criteria as today until v3.15.7.** |
| `vol_compression_breakout_crypto_1h` | strict | `exploratory` | active_discovery hypothesis. **Classification change only — same screening criteria as today until v3.15.7.** |
| `crypto_diagnostic_1h` | diagnostic | `exploratory` | diagnostic_only + excluded from candidate promotion. **Classification change only — same screening criteria as today until v3.15.7.** |

Important: the three `exploratory` assignments are **classification
change only**. They do NOT alter screening thresholds, gates, or
criteria. Any screening-criteria difference between phases must be
introduced in v3.15.7 with explicit hypothesis evidence.

No current preset uses `standard`. The value is reserved for a
future research middle layer; it is functionally available today
and validates cleanly.

## 4. Propagation chain

```
preset.screening_phase
        │
        ▼
run_research.run_research(...)
   ├─ _enforce_preset_validation(preset_obj, tracker)
   │     (raises PresetValidationError under strict mode on
   │      screening_phase_invalid)
   │
   ├─ tracker.emit_event("screening_phase_active",
   │                     preset_name=..., screening_phase=...)
   │
   ├─ build_run_meta_payload(... screening_phase=preset.screening_phase)
   │     → research/run_meta_latest.v1.json (schema_version "1.2")
   │
   └─ for each candidate:
        tracker.emit_event("screening_phase_observed",
                           candidate_id=..., screening_phase=...)
        execute_screening_candidate_isolated(
            ...,
            screening_phase=preset_obj.screening_phase,
        )

batch_execution.execute_screening_batch(...):
   - has NO preset, NO tracker in scope
   - passes screening_phase=None explicitly (no inference)
   - emits NO tracker event
```

Key non-goals:

- **`batch_execution` does not infer the phase.** It must NOT
  derive ``screening_phase`` from ``screening_mode`` /
  ``preset_class`` / ``hypothesis_id`` / diagnostic flags. The
  function signature is unchanged; it simply passes ``None``.
  Pinned by `tests/unit/test_v3_15_6_batch_execution_no_inference.py`.
- **`screening_runtime.py` is unchanged.** v3.15.6 stops at the
  `screening_process` boundary. v3.15.7 may pass the phase deeper
  if needed.
- **`execute_screening_candidate_isolated` does not branch on the
  phase** — the kwarg is accepted and discarded via
  `del screening_phase`. The result dict is **not** extended;
  this prevents stealth schema drift through
  `runtime_record.update(outcome)` on
  [research/batch_execution.py:191](research/batch_execution.py:191).

## 5. Validation

| Mode | Default (no env var) | Strict (`QRE_STRICT_PRESET_VALIDATION=1`) |
|---|---|---|
| valid `screening_phase` | run continues | run continues |
| invalid `screening_phase` | tracker event `preset_validation_warning`; run continues | `PresetValidationError` raised |

`PresetValidationError` is **not** a `DegenerateResearchRunError`.
Under strict mode an invalid phase causes the CLI to exit with
rc=1 (Python's default for an uncaught exception), which the
v3.15.5 launcher dispatch maps to outcome `"technical_failure"`
— never `research_rejection`, `degenerate_no_survivors`, or
deprecated `worker_crashed`.

Tests pin both paths:

- `tests/unit/test_v3_15_6_validation_warning_path.py`
- `tests/unit/test_v3_15_6_validation_strict_path.py`
- `tests/integration/test_v3_15_6_invalid_phase_technical_failure.py`

  `tests/unit/test_v3_15_6_run_meta_schema.py::test_v11_payload_without_screening_phase_remains_readable`.

## 8. v3.15.7 forward path

When v3.15.7 introduces phase-aware criteria, the seam is
already in place:

1. ``execute_screening_candidate_isolated`` accepts
   ``screening_phase: str | None`` (not Literal — leaves room for
   vocab extension).
2. The signature is publicly importable and pinned by
   `tests/unit/test_v3_15_6_v3_15_7_compatibility_seam.py`.
3. v3.15.7 may dispatch criteria inside the function body or
   inside `_build_child_payload` / `execute_screening_candidate_samples`
   without API change.
4. v3.15.7 may add new tracker event payload fields (already
   open) and may extend the run_meta schema again
   (1.2 → 1.3) following the same backward-compat discipline.
5. v3.15.7 may pass the phase deeper into `screening_runtime.py`
   if needed; v3.15.6 left that module ungewijzigd by design.

What v3.15.7 should NOT do:

- Rename ``screening_phase`` (it's now public surface in run_meta
  and tracker events).
- Reuse the legacy ``screening_mode`` for criteria dispatch.
- Mutate frozen contracts.
