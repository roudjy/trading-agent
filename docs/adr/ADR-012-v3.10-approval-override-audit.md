## Context

v3.8 and earlier phases introduced diagnostic instrumentation into
the research execution path. The v3.10 scope calls for explicit
isolation or removal of any "approval override" style logic in
`engine.py` so it cannot silently act as production promotion
behavior.

The v3.10 blueprint named `engine.py` specifically. A preliminary
grep of `agent/backtesting/engine.py` for `approval`, `override`,
`bypass`, `force`, `skip_gate`, and `approved` returned no hits
(the only matches in the research tree are
`research/promotion.py::_merge_overrides`, which is user-config
overrides for promotion gate *thresholds* and not a gate bypass,
and a comment in `research/results.py` about "approved contract
change" which is unrelated).

## Audit task

The Fase 1 audit commit runs the final grep and this ADR is updated
with one of three outcomes:

1. **Outcome A — no production bypass found.** The ADR records the
   grep scope and the result. No code change.
2. **Outcome B — diagnostic bypass found.** The ADR documents the
   location, what it bypassed, and mandates gating behind an
   explicit `--diagnostic` CLI flag plus a `diagnostic_only=True`
   field in the run-meta sidecar. Runs with that flag cannot
   promote.
3. **Outcome C — production bypass found.** The ADR documents the
   removal patch and marks the superseded behavior.
