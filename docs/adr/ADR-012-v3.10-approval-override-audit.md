# ADR-012: v3.10 Approval-Override Audit

Status: Accepted
Date: 2026-04-22
Phase: v3.10 (hardening cleanup, pre-UI blocker)
Supersedes: -
Superseded by: -
Related: ADR-011 (v3.10 architecture), ADR-009 (platform layer)

## Context

v3.8 and earlier phases introduced diagnostic instrumentation into
the research execution path. The v3.10 scope calls for explicit
isolation or removal of any "approval override" style logic in
`engine.py` so it cannot silently act as production promotion
behavior.

The v3.10 blueprint named `engine.py` specifically. A preliminary
grep of `agent/backtesting/engine.py` for `approval`, `override`,
`bypass`, `force`, `skip_gate`, and `approved` returned no hits
(the only matches in the research tree are
`research/promotion.py::_merge_overrides`, which is user-config
overrides for promotion gate *thresholds* and not a gate bypass,
and a comment in `research/results.py` about "approved contract
change" which is unrelated).

## Audit task

The Fase 1 audit commit runs the final grep and this ADR is updated
with one of three outcomes:

1. **Outcome A — no production bypass found.** The ADR records the
   grep scope and the result. No code change.
2. **Outcome B — diagnostic bypass found.** The ADR documents the
   location, what it bypassed, and mandates gating behind an
   explicit `--diagnostic` CLI flag plus a `diagnostic_only=True`
   field in the run-meta sidecar. Runs with that flag cannot
   promote.
3. **Outcome C — production bypass found.** The ADR documents the
   removal patch and marks the superseded behavior.

## Audit scope

Files grepped for `approval`, `override`, `bypass`, `force`,
`skip_gate`, `manual_approve`, `auto_promote`, `force_approve`,
`approved`:

- `agent/backtesting/engine.py`
- `agent/backtesting/*.py` (siblings)
- `research/promotion.py`
- `research/run_research.py`
- `research/registry.py`
- `orchestration/**/*.py`

## Audit findings (2026-04-22)

Grep of the full audit scope returned only false-positives:

- `agent/backtesting/cost_sensitivity.py:5,99` — "slippage override" and
  "fee multiplier override" are scenario parameters in the evaluation
  replay layer (v3.8 step 3). They mutate the *scenario*, not a
  production gate. Diagnostic-only; already opt-in via
  `BacktestEngine.build_cost_sensitivity` and not called from `run()`.
- `research/results.py:11` — "Do NOT edit without an approved contract
  change" is a comment on the frozen ROW_SCHEMA tuple. No code path.
- `research/run_research.py:2008,2500` — the two "no longer bypasses
  the Orchestrator" comments document that v3.9 phase 4 *removed* the
  previous inline-mode bypass of `Orchestrator.dispatch_serial_batches`.
  The bypass no longer exists; the comments are historical reference
  for code reviewers.
- `research/promotion.py:27` — `normalize_promotion_config` merges
  *user-provided thresholds* with defaults. It does not permit
  skipping `classify_candidate`, `_check_rejection_rules`, or any gate
  evaluation. The keys it accepts are numeric/flag thresholds
  (e.g. `min_oos_trades`, `require_leakage_checks_ok`), not bypass
  toggles. `require_goedgekeurd: False` is the inherited default and
  keeps engine-side goedkeuring *optional*, not bypassed — the
  research-layer promotion gates still run in full.
- No matches at all in `orchestration/`.

## Decision

**Outcome A — no production bypass is present in the current codebase.**

No code change is required. `engine.py` contains no approval-override
logic. `promotion.py::normalize_promotion_config` is a threshold-merge
function and is within the AGENTS.md-sanctioned user-config surface.

If any future commit reintroduces a gate-bypass path, it MUST:

1. Be gated behind an explicit `--diagnostic` CLI flag on
   `run_research.py`.
2. Set `diagnostic_only=True` on the run-meta sidecar
   (`research/run_meta_latest.v1.json`).
3. Be rejected by the promotion layer via the
   `excluded_from_candidate_promotion` check introduced in v3.10
   (see ADR-011 §9).

## Consequences

- The v3.10 hardening suite ships the strategy-name precondition,
  screening-interrupt reason_detail regression, and ExecutionEvent
  roundtrip regression as the observable guarantees that the
  no-bypass invariant holds at the data-integrity layer.
- Future ADRs introducing diagnostic paths must cite this ADR and
  the three requirements above.
- No additional action required for the v3.10 release.
