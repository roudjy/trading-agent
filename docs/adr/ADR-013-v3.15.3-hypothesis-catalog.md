# ADR-013 — v3.15.3 Strategy Hypothesis Catalog

Status: **Accepted** — 2026-04-25
Branch: `feature/v3.15.3-hypothesis-catalog`

## Context

v3.15.2 shipped the Autonomous Campaign Operating Layer (COL): an
hourly `campaign_launcher` tick that selects a single template per
tick and routes it through Filter 0 (`_check_template_eligibility`)
plus the existing v3.15.2 budget / family / preset / lease /
duplicate gates. Production verification (2026-04-25) showed R0 /
R1 / autonomous spawn all working end-to-end.

What v3.15.2 lacked: a **first-class artifact-backed concept of a
strategy hypothesis**. The COL could enable / disable an entire
preset, but it couldn't reason about whether the strategy bundled
inside that preset was *worth running* against the current evidence
landscape. Adding more strategies meant either:

1. Adding more presets (which conflates hypothesis ≈ preset and
   inflates the campaign template catalog), or
2. Cramming multiple hypotheses into one preset bundle (which
   defeats the v3.15.2 selection-by-preset model).

We needed a closed-enum status surface — `active_discovery /
planned / disabled / diagnostic` — that the campaign policy could
read at tick boundaries to gate spawning. With exactly **one**
`active_discovery` hypothesis we keep the COL focused on a single
controlled experiment at a time, preserving the system's character
as an alpha-filter rather than alpha-generator.

## Decision

Introduce a Strategy Hypothesis Catalog as a tuple of frozen
`StrategyHypothesis` rows in `research/strategy_hypothesis_catalog.py`
plus a parallel `research/strategy_campaign_metadata.py` carrying
campaign-policy fields (eligible types, cooldown, followup, priority,
failure-mode mapping). Both write adjacent sidecars on every research
run via the existing `_sidecar_io.write_sidecar_atomic` + the v3.15.2
`campaign_os_artifacts.build_pin_block` helper.

A new `EligibilityPredicate.require_hypothesis_status: tuple[str, ...]`
field — defaulting to `()` — extends the v3.15.2 frozen template
contract. `EligibilityPredicate.to_payload()` **omits** the field when
empty so the existing 3 baseline preset templates remain bytewise
identical in `campaign_templates_latest.v1.json`.

The campaign policy's Filter 0 (`_check_template_eligibility`) gains
a fifth check after `require_preset_status`: bridge
`preset.bundle[0] -> registry.strategy_family -> catalog.status`,
reject with one of `hypothesis_not_in_catalog`,
`hypothesis_status_<status>_not_in_required`,
`strategy_not_in_registry`, or `preset_bundle_empty`.

Trend Pullback is the only `active_discovery` hypothesis. A new thin
strategy `trend_pullback_v1_strategie` (max 3 parameters), a new
composite feature `pullback_distance`, a new registry entry
`trend_pullback_v1` (`strategy_family="trend_pullback"`), and a new
preset `trend_pullback_crypto_1h` (bundle one strategy) carry it
through to the COL.

## Why a separate failure taxonomy module

The v3.11 `research/rejection_taxonomy.py` owns the
**promotion-layer** rejection codes — pinned by candidate registry v2
and the v3.11/v3.13 ADR contract. The campaign-layer needs a
**different** vocabulary (cost_fragile, parameter_fragile, overtrading,
…) describing why a *campaign* failed, not why a *candidate* was
rejected from promotion. Stuffing both in the same module would risk
breaking the v3.11 contract every time we evolve the campaign-side
codes. So `research/strategy_failure_taxonomy.py` lives as an
adjacent module with a closed canonical-codes tuple plus
strategy-specific `STRATEGY_SPECIFIC_ALIASES` (e.g.
`trend_pullback_cost_fragile -> cost_fragile`) and a typed
`canonicalize()` helper that loud-fails on unknown codes.

## Why ``trend_pullback_v1`` instead of replacing legacy ``trend_pullback``

The legacy `trend_pullback` (6 parameters) and `trend_pullback_tp_sl`
(8 parameters) were registered pre-v3.5 with `strategy_family =
trend_following` and use the legacy non-thin `func(df)` contract.
They are referenced by the optional bundle of
`trend_equities_4h_baseline` and may be used by archived candidate
registry rows. Replacing them in-place would break those references.

`trend_pullback_v1` is registered alongside as a separate strategy
with `strategy_family = "trend_pullback"` (unique to v3.15.3). The
catalog row `trend_pullback_v1` bridges only to that new family;
the legacy entries stay in `trend_following` family and are
intentionally NOT part of the v3.15.3 hypothesis-catalog bridge.

This preserves backward compatibility (legacy strategies remain
runnable in their original presets and any archived audit trails)
while keeping the v3.15.3 hypothesis catalog clean: exactly one
active_discovery row mapping to exactly one thin strategy.

## Why default ``require_hypothesis_status = ()``

The v3.15.2 `campaign_templates_latest.v1.json` sidecar has been
written every hour since the production cutover. Downstream
consumers (frontend, audit tooling, archived snapshots) parse it
with the v3.15.2 schema. Adding a new field to every existing
template would silently break those consumers.

Defaulting the field to `()` plus omitting it from `to_payload()`
when empty keeps the existing 3 baseline preset templates
**bytewise identical**. Only the new `trend_pullback_crypto_1h`
templates carry `require_hypothesis_status=("active_discovery",)`,
and they are explicitly new entries — consumers that surfaced them
will surface them with the new field, no schema migration needed.
Pinned by `tests/regression/test_v3_15_2_campaign_templates_byte_identity.py`.

## Out of scope

- Auto-status transitions (`active_discovery → cooldown / negated /
  deprioritized` based on accumulated campaign evidence). v3.15.3
  ships hand-curated initial statuses; future versions will let the
  campaign launcher propose status changes via a new event type.
- Multi-strategy hypothesis bundles. The bridge uses
  `preset.bundle[0]` only; multi-strategy bundles continue to be
  gated only by the v3.15.2 preset filters with
  `require_hypothesis_status = ()`.
- Executable implementations of `atr_adaptive_trend`,
  `volatility_compression_breakout`, or `dynamic_pairs`. They appear
  only as metadata rows.
- ML/RL selector. The catalog is a hand-curated configuration
  surface, not a black-box scorer.
- Live / shadow enablement. All artifacts pin
  `live_eligible=False` (campaign_os v0.1 invariant).

## Consequences

- The COL can autonomously refuse to spawn `planned`, `disabled`, or
  `diagnostic` hypotheses with a canonical reject reason that is
  recorded in `campaign_policy_decision_latest.v1.json` for audit.
- Adding a sixth strategy is now a two-line catalog change plus a
  new preset; the v3.15.2 templates layer scales by hypothesis status
  rather than by preset count.
- The frozen v3.5+ public output contracts (`research_latest.json`,
  `strategy_matrix.csv`) are untouched — the catalog is adjacent.
- The v3.11 `rejection_taxonomy` and the v3.15.2 `campaign_templates`
  byte-identity for legacy presets are explicitly preserved by
  regression tests.
