# ADR-011: v3.10 Architecture — Research Ops & Frontend Migration

Status: Accepted
Date: 2026-04-22
Phase: v3.10 (research ops + frontend migration)
Supersedes: -
Superseded by: -
Related: ADR-009 (platform layer introduction), ADR-010 (v3.9 closure)

## Context

v3.9 closed the orchestration platform. The research path is now
executable end-to-end, but day-to-day operations still require a
copy-paste loop between terminal, Claude, and GPT. There is no
first-class preset catalog, no single-command ops entrypoint, no
automatic daily run, no dashboard anti-indexing, and no controlled
deploy channel from `main` to the VPS.

v3.10 closes the ops gap. It introduces preset-driven research,
a `researchctl` CLI, a post-run analysis/report agent, a React +
TypeScript SPA on `:8050`, and explicit, controllable VPS
deployment. The layer boundaries (`research/` ⟂ `agent/backtesting/`
⟂ `orchestration/`) stay intact, the public output contracts
(`research_latest.json`, `strategy_matrix.csv`) remain frozen, and
no alpha expansion is introduced.

## Decisions

### 1. Flask stays as backend / API runtime

The existing Flask app in `dashboard/dashboard.py` remains the
control-surface runtime. Auth (session + operator-token), the 25+
existing `/api/*` endpoints, and `research_runner.launch_research_run`
are untouched except for additive extensions. Only the Flask-rendered
templates (`dashboard.html`, `research_control_surface.html`) are
replaced by a React SPA.

Rejected: rewriting to FastAPI. It would duplicate working auth, risk
breaking the existing operator-token HMAC path, and add dependencies
without enabling anything in scope.

### 2. React + TypeScript SPA served by Flask

The SPA lives under `frontend/` as a Vite + React 18 + TS project.
The build artifact (`frontend/dist/`) is copied into the Python
runtime image via a multi-stage Dockerfile and served by Flask as
static files (`/` → `index.html`, `/assets/*` → `dist/assets`).

The legacy templates stay reachable at `/legacy/dashboard` and
`/legacy/research-control` for one release as a safety net.

Rejected: a separate frontend container. Two containers for a single
static bundle is unnecessary complexity on this VPS.

### 3. nginx reverse proxy for anti-indexing

A new `nginx` service binds host `:8050` and reverse-proxies to the
internal `dashboard:8050`. It injects `X-Robots-Tag: noindex, nofollow,
noarchive, nosnippet` globally, serves `robots.txt` with
`Disallow: /`, and blocks known AI/crawler User-Agents with HTTP 403.

Rejected: Flask-only header injection. It cannot block by UA cleanly
at the HTTP layer and would mix request-filtering with application
logic.

### 4. Deploy via explicit script, no watchtower

Deployment from GHCR `:latest` to the VPS goes through
`scripts/deploy.sh` (git pull + `docker compose pull` + `up -d` +
health check) combined with `docker-compose.prod.yml` that pulls
`image: ghcr.io/.../trading-agent-*:${IMAGE_TAG:-latest}`.

`researchctl` does **not** ship a `deploy` subcommand. Deploy is
strictly an ops concern and lives in `scripts/deploy.sh`, invoked
manually on the VPS (or by CI once a VPS SSH secret is added —
out of scope for v3.10).

Rejected: Watchtower. Auto-pulling `:latest` tags mid-run increases
risk of restart during a live research job and reduces operator
control. The explicit script keeps deploys observable, reviewable,
and reversible via `IMAGE_TAG=<prev-tag> bash scripts/deploy.sh`.

### 5. Daily scheduling via systemd-timer on the VPS host

The default daily research run (`trend_equities_4h_baseline`) is
triggered by a host-level systemd-timer that calls
`docker exec jvr_dashboard python /app/researchctl.py run
trend_equities_4h_baseline` at 06:00 UTC. Unit + timer files ship
in `ops/systemd/` with an install README.

Rejected: an in-container Python scheduler daemon, or a compose
`scheduler` service. A container-internal daemon adds a moving part
and duplicates state that systemd already provides (next-fire,
persistence across boots, journal). Host systemd is the simplest
architecturally correct location for this.

The systemd-unit skips preset 4 (`crypto_diagnostic_1h`) explicitly;
it is a diagnostic-only preset and must never auto-run.

### 6. Preset 1 is small and orthogonal

`trend_equities_4h_baseline` bundles two independent edges:
`sma_crossover` (trend-following baseline) and `breakout_momentum`
(breakout baseline). `trend_pullback` is available as an optional
addition but is not part of the default daily bundle. Management
variants (e.g. `trend_pullback_tp_sl`) and stubs (`atr_breakout_STUB`)
are kept out to prevent strategy-explosion and alpha drift.

### 7. Preset 2 is planned / disabled in v3.10

`pairs_equities_daily_baseline` is registered with `enabled=False`
and `status="planned"`. Reason: the current `pairs_zscore` is
bytewise-pinned under v3.6 scope-lock with `reference_asset="ETH-EUR"`
and `hedge_ratio=[1.0]` hardcoded — architecturally suited to a
fixed-hedge crypto pair, not to equity pairs that need OLS hedge
ratio and multi-reference-asset support. Enabling it now would be
scope drift. `backlog_reason` points to a v3.11 equity-pairs ADR.

### 8. Preset 3 uses regime as a FILTER, not as an alpha

`trend_regime_filtered_equities_4h` primary bundle is the trend
strategies (`sma_crossover`, `breakout_momentum`). The
`bollinger_regime` logic is reused as a pre/post screening filter
(`regime_filter="bollinger_regime_derived"`, modes `trend_only` and
`low_vol_only`). No bollinger-based mean-reversion strategy is
primary in this preset. No new registry entry is created; the
filter lives in preset configuration only.

### 9. Preset 4 is explicitly excluded from promotion and scheduling

`crypto_diagnostic_1h` carries three hard flags:
`diagnostic_only=True`, `excluded_from_daily_scheduler=True`,
`excluded_from_candidate_promotion=True`. The promotion layer reads
the run-meta sidecar and refuses to promote candidates from runs
carrying that flag; if the sidecar is missing, the safe default is
"excluded".

### 10. Strategy-name / data-integrity fix is the first hardening commit

The `make_result_row` precondition (raise on `strategy["name"]`
None/empty) is the blocker gate for every other v3.10 workstream.
Regression tests for screening-interrupt reason_detail and
ExecutionEvent JSON roundtrip complete the hardening suite. The
UI, preset, CLI, and ops work do not start until
`bash tests/run_tests.sh` is green.

### 11. Public output contracts remain frozen

`research/results.py` `ROW_SCHEMA`, `JSON_TOP_LEVEL_SCHEMA`, and
`JSON_SUMMARY_SCHEMA` are byte-identical to pre-v3.10. New v3.10
fields (preset name, hypothesis, universe, bundle, screening_mode,
cost_mode, completed_at_utc, top rejection reasons rollup,
candidate summary) land in a new adjacent sidecar at
`research/run_meta_latest.v1.json`. The daily report is written to
`research/report_latest.md` + `research/report_latest.json` (also
new, also adjacent).

### 12. `/api/research/run-status` remains canonical

The existing endpoint at `dashboard/dashboard.py:266` is the single
source of truth for run status in the React UI. No new
`/api/run-status` is introduced.

## Consequences

- Operators run `bash scripts/deploy.sh` on the VPS after a main
  merge. There is no implicit auto-deploy.
- The systemd-timer must be enabled once per host
  (`systemctl enable --now trading-agent-daily-research.timer`).
  `scripts/deploy.sh` checks and reports unit status.
- Adding equity-pairs support (OLS hedge, multi-reference-asset)
  is deferred to v3.11 via a new ADR.
- The ATR / volatility-breakout strategy is deferred; the preset
  resolver tolerates missing bundle entries with a `backlog_reason`.
- The SPA runs on HTTP without `Secure` cookies. TLS is out of
  scope for v3.10.

## Non-choices

- No watchtower.
- No compose-scheduler service.
- No `researchctl deploy` subcommand.
- No new public-contract fields in `research_latest.json` or
  `strategy_matrix.csv`.
- No strategy-registry expansion for preset 3's regime filter.
