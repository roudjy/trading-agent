# ADR-010: v3.9 Closure — Accepted Debt and Deferred Items

Status: Accepted
Date: 2026-04-22
Phase: v3.9 (orchestration hardening, phase 6 — closure)
Supersedes: -
Superseded by: -
Related: ADR-009 (platform layer introduction)

## Context

v3.9 was scoped as orchestration hardening on top of a proven,
pinned v3.8 engine. The phase plan was:

- Phase 1: package shell and boundary contract (`orchestration/`,
  dependency lint, ADR-009).
- Phase 2: Task data model (`Task`, `TaskResult`, `TaskFailure`,
  `ReasonCode`, retriable classification).
- Phase 3: `ExecutionBackend`, `Orchestrator` as a thin
  behavior-preserving seam.
- Phase 4: `Scheduler`, `TaskQueue`, unified dispatch ownership for
  inline and parallel paths; cross-Batch parallelism preserved
  (with bytewise evidence).
- Phase 5: Failure taxonomy via `BatchOutcome`, explicit typed
  outcome contract, artifact-truth invariant tests.
- Phase 6: Final closure — crash/resume canary, fallback-scope
  hardening, explicit debt decisions, formal exit assessment.

Phase 6 is the closure gate. Its purpose is to answer "is v3.9
operationally reliable enough to formally close the build phase
and move into research-first work?" The answer this ADR records
is: **yes, with the accepted debt enumerated below**.

## Decision

v3.9 is formally complete. Closure rests on five pillars:

1. **Bytewise determinism preserved.** Tier 1 digest pins (SMA
   crossover, z-score mean reversion, pairs z-score, multi-asset
   parity, fitted-feature) all pass unchanged. Public artifact
   schemas (`research_latest.json`, 19-column CSV,
   integrity/falsification sidecars, run/batch/candidate state
   artifacts) bytewise unchanged.
2. **Orchestration is a real, bounded layer.** `orchestration/` is
   a peer package to `agent/`, `research/`, `dashboard/`. The
   dependency-direction lint (`tests/unit/test_orchestration_boundary.py`)
   enforces the allowed-import matrix at CI time. Production code
   in `research/run_research.py` supplies explicit hooks; no
   silent dependency on fallback translation (pinned by
   `tests/unit/test_orchestration_default_complete_scope.py`).
3. **Failure semantics are typed and observable.** `BatchOutcome`
   is the explicit outcome contract. Dispatch drives
   `stop_on_failure` from the outcome, not from research-semantic
   dict state. `is_retriable()` is observable (not yet acted on).
4. **Artifact-truth is structurally guaranteed.** The Queue is an
   in-memory cache bound to one Orchestrator instance. Recovery
   does not import orchestration. Task IDs are deterministic.
   Pinned by `tests/unit/test_orchestration_artifact_truth.py`.
5. **Recovery works end-to-end.** A Phase-6 canary
   (`tests/resilience/test_orchestration_crash_resume_canary.py`)
   forces a mid-run crash, re-invokes `run_research(resume=True,
   retry_failed_batches=True)`, and validates that the run
   completes with no duplicate results, all batches terminal, and
   a consistent lifecycle status.

## Accepted debt (explicit decisions)

The following were raised during Phase 2–5 review and are
explicitly classified here. Each item carries one of:
**resolved-now**, **accepted**, or **deferred-to-v4+**.

### 1. `TaskResult.payload["batch_result"]` — loose dict shape

**Classification:** **accepted debt**.

**Rationale.** `TaskResult.payload` is a `Mapping[str, Any]` that
carries the raw `execute_screening_batch` /
`execute_validation_batch` result dict from worker to hook. The
dict is already shape-validated inside research (via
`_merge_screening_batch_result` / `_merge_validation_batch_result`),
so an orchestration-level typed wrapper would be ceremony without
safety gain today. v3.9's crash/resume canary exercises this path
end-to-end without incident; no failure / retry / resume decision
turns on the dict's shape.

**When to revisit.** If a later phase introduces a second consumer
of the payload (e.g., a platform event log or cross-run analytics),
typing this surface becomes worth it. Until then, the loose dict is
a permitted temporary contract and the `_merge_*_batch_result`
functions retain sole shape-validation responsibility.

### 2. Closure-heavy inline adapters in `research/run_research.py`

**Classification:** **accepted debt**.

**Rationale.** The inline `_inline_screening_batch` and
`_inline_validation_batch` closures capture ~20 outer-scope
variables each. They are called exactly once per batch and are
exercised by the full integration test suite plus the Phase-6
crash/resume canary. There is no Phase-6 correctness concern that
a refactor would close. The closures preserve the v3.8 per-batch
body bytewise.

**When to revisit.** If a later phase introduces batch-level retry
or per-candidate dispatch that requires re-invoking the inline body
with a different scope, the closures will need to be extracted.
That is a v4+ concern and is not blocking v3.9 closure.

### 3. No automatic intra-run retry

**Classification:** **deferred-to-v4+**.

**Rationale.** `BatchOutcome.is_retriable()` and the
`RETRIABLE_REASONS` / `NON_RETRIABLE_REASONS` partition are in
place. The dispatch does not act on them: a retriable failure
still stops dispatch with `stop_on_failure=True`, exactly like a
non-retriable failure. This is intentional — the research engine
is deterministic in its inputs, so retrying a failed task on the
same inputs yields the same failure. The retry surface exists as
preparation for v4 flaky-transient cases (network timeouts on
data loading, OOM-killed workers, etc.) where retry is
meaningfully useful.

**When to revisit.** When v4 introduces genuinely transient
failure surfaces (e.g., distributed workers, remote data
providers) or when a specific existing failure mode is proven
flaky enough to warrant bounded retry. Any retry implementation
must include attempt-id encoding in the Task (which the existing
`task_id` contract already supports), a bounded max-attempt
count, and tests for each retriable reason code.

### 4. Coarse worker-side exception classification

**Classification:** **accepted debt**.

**Rationale.** Workers currently propagate exceptions unchanged.
The dispatch's `future.result()` call raises; the runner's
top-level try/except records the failure in the lifecycle
artifacts. No exception type is translated into `ReasonCode` at
the worker boundary. The hook-supplied `BatchOutcome` is where
reason-code classification happens, based on inspection of
post-merge batch state.

This works because the engine's failure surface is narrow:
`FoldLeakageError` is a deterministic correctness violation that
should propagate to abort the run; other exceptions are effectively
`STRATEGY_ERROR` by conservative default (pinned by
`classify_batch_reason`).

**When to revisit.** If `ExecutionEvent` v1.1 introduces richer
emission kinds (partial_fill, rejected, canceled — ADR-008
scaffolded but unused) and a worker wants to translate those
into typed `TaskFailure` results, the worker-side translation
becomes real work. Until then, coarse classification is fine.

### 5. `_default_complete` fallback

**Classification:** **resolved now (Phase 6)**.

**What changed.** The docstring explicitly scopes it to "legacy
/ test fallback only". Every `dispatch_serial_batches` /
`dispatch_parallel_batches` call in `research/run_research.py`
now supplies an explicit `on_batch_complete=` hook. Production
code never exercises the default. A static AST test
(`tests/unit/test_orchestration_default_complete_scope.py`) pins
the rule.

**Why not delete it entirely.** Unit tests that construct
`Orchestrator` directly rely on the default for convenience.
Deleting it would force mechanical churn across unrelated tests
with no correctness benefit. The scope-pin test catches future
production drift; the default stays as a documented test aid.

### 6. Cross-Batch parallelism bytewise-equivalence evidence

**Classification:** **accepted debt with caveat documented**.

**Rationale.** Phase 4's bytewise-equivalence test
(`tests/integration/test_orchestration_bytewise_equivalence.py`)
compares `max_workers=1` and `max_workers=4` output using
`ThreadPoolExecutor` (monkey-patched) rather than
`ProcessPoolExecutor`. ThreadPoolExecutor is what lets the
fixture's monkey-patched strategies remain visible to workers.
True production config uses `ProcessPoolExecutor`.

The gap: the evidence is strong for "the Orchestrator's
rolling-submit + ordered-collection pattern is bytewise
deterministic" and for "the engine is deterministic in its
inputs"; combined, those imply `ProcessPoolExecutor`
equivalence. But the direct test does not exercise
`ProcessPoolExecutor` with production strategies and production
data.

**When to revisit.** When v4 introduces a distributed or
durable-queue backend, the bytewise-equivalence test should be
extended to cover the new backend. A production-scale canary
(real strategies, real data, both `max_workers=1` and
`max_workers=N`) is also worth running as a pre-release check
for v4.

## Consequences

### Preserved bytewise

- Tier 1 digest pins (SMA, z-score MR, pairs z-score, multi-asset
  parity, fitted-feature).
- `research_latest.json` row and top-level schema.
- 19-column CSV row schema.
- Integrity and falsification sidecar schemas.
- Run manifest, run state, batch state, candidate state.
- Screening candidate sidecar.
- `candidate_id` hashing inputs.
- Walk-forward `FoldLeakageError` semantics.
- Engine purity (no concurrency primitives added; no
  orchestration imports).

### Added (across all of v3.9)

- `orchestration/` top-level package: `__init__.py`, `task.py`,
  `executor.py`, `worker.py`, `orchestrator.py`, `queue.py`,
  `scheduler.py`.
- `orchestration/__init__.py` public API exports: `Task`,
  `TaskResult`, `TaskFailure`, `ReasonCode`, `BatchOutcome`,
  `OutcomeKind`, `Orchestrator`, `Scheduler`, `FifoScheduler`,
  `TaskQueue`, `TaskQueueError`, `TaskKind`, `ExecutionBackend`,
  `InlineBackend`, `ProcessPoolBackend`, `TaskOutcome`,
  `RETRIABLE_REASONS`, `NON_RETRIABLE_REASONS`, `is_retriable`,
  `build_task_id`, `classify_batch_reason`, `deepcopy_batch`,
  `ORCHESTRATION_LAYER_VERSION`.
- Static boundary-enforcement test
  (`tests/unit/test_orchestration_boundary.py`).
- Explicit failure-taxonomy tests
  (`tests/unit/test_orchestration_failure_taxonomy.py`).
- Artifact-truth invariant tests
  (`tests/unit/test_orchestration_artifact_truth.py`).
- Bytewise-equivalence integration test between inline and
  parallel dispatch
  (`tests/integration/test_orchestration_bytewise_equivalence.py`).
- Crash/resume canary
  (`tests/resilience/test_orchestration_crash_resume_canary.py`).
- `_default_complete` scope-pin
  (`tests/unit/test_orchestration_default_complete_scope.py`).
- ADR-009 (platform layer introduction) and this ADR-010 (v3.9
  closure).
- v3.9 addendum in `docs/orchestrator_brief.md` describing the
  orchestration layer.

### Deferred (v4+)

- Automatic intra-run retry.
- Distributed execution / cross-host orchestration.
- Persistent/durable queue as source of truth (the Queue remains
  in-memory cache only).
- Task-level resume (beyond the existing batch-level path).
- Dashboard launch API upgrade.
- Campaign framework spanning multiple runs.
- `ExecutionEvent` richer emission (partial_fill, rejected,
  canceled).
- Typed `BatchTaskResult` wrapper replacing the loose
  `payload["batch_result"]`.
- Selective file relocation of orchestration-native `research/`
  modules into `orchestration/` (`research/run_state.py`,
  `research/observability.py::ProgressTracker`,
  `research/recovery.py`).

## Rejected alternatives during v3.9

- **Engine-internal orchestration.** Rejected in ADR-009.
- **Separate repo or microservice for orchestration.** Rejected in
  ADR-009.
- **Package name `platform/`.** Rejected in ADR-009 due to the
  stdlib `platform` shadow.
- **Large Phase-2 relocation wave.** Rejected in the Phase-1
  revised design review; replaced with "capability first,
  relocation later, zero to two moves total". Actual v3.9
  relocations: zero.
- **Automatic retry in Phase 5.** Rejected as premature. Deferred
  to v4+.
- **Task-level resume in Phase 5.** Rejected as invasive.
  Deferred to v4+.
- **Platform event log in Phase 5.** Rejected as duplicative of
  `ProgressTracker.emit_event`.
- **Eradicating `_default_complete` in Phase 6.** Rejected as
  unnecessary churn; scoped it to test-only via docstring + AST
  pin instead.

## v3.9 formal closure assertion

v3.9 is **closed** as of Phase 6. The exit criteria are:

1. ✅ Full suite green (baseline 827 + Phase 1–6 additions, zero
   regressions).
2. ✅ Tier 1 bytewise pins unchanged.
3. ✅ Public artifact contracts bytewise unchanged.
4. ✅ Engine purity intact.
5. ✅ Orchestration layer boundary enforced by CI lint.
6. ✅ Failure taxonomy typed and observable.
7. ✅ Artifact-truth invariant structurally guaranteed and tested.
8. ✅ Crash/resume canary green, demonstrating end-to-end recovery
   on an interrupted run.
9. ✅ `_default_complete` fallback scoped and pinned.
10. ✅ Remaining debt explicitly classified (this ADR).

The project is ready to move from the v3.9 build phase into
research-first work, with scoped v4 evaluation as a separate
decision gated by concrete research need.
